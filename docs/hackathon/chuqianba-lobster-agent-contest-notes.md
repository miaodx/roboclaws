## 官方速览

### 比赛介绍

出钳大舞台，有想法你就来。

参赛对象和工具范围很开放：

- 无论是个人虾还是岗位虾，甚至是 Codex、Claude Code 等工具。
- 无论是业务人员还是产品研发。
- 无论现在只是想法雏形，还是已经落地跑通。
- 无论想一人 FDE，还是团队作战脑暴。

只要具备业务价值，都可以展示想法。未来，“出钳吧！”将作为常态化赛事开展。

### 比赛意义

官方速览里强调四个意义：

1. 低门槛，广撒网：打破“只有写代码的人才能参赛”的门槛。无论是业务人员、产品经理还是研发工程师，无论使用龙虾、Codex、Claude Code 还是其他 Agent 工具，只要在真实工作中跑出了业务价值，甚至当前只是一个好 idea，都可以来展示。
2. 发掘 Agent 时代的新型业务创造者：识别一批能够主动结合 AI Agent 能力重构工作流、优化业务流程、提升组织效率的“超级个体”。
3. 沉淀可复制的资产：把散落在各业务线的优秀实践提炼成可复制的 Agent 解决方案和养虾心得。所有入围案例进入 AI 需求池，赛后优先提供落地支持，让好想法不止停在比赛里。
4. 为公司 AI 战略提供一线炮火：出钳吧产出的高价值案例，将直接输入发布会样板库与黑客松共建场景，让一线真实需求与最佳实践成为公司 AI 投入方向的重要参考依据。

1. 提报人需为小米集团正式员工。
2. 提报案例必须是基于 Agent 进行的真实场景案例。若需要个人虾等资源申请，可走“绿通”，各战区根据所领报名数量指标开虾配额。

入围和复赛阶段：

- 形式要求：文档即可；若有 demo 可插入文档。
- 内容要求包括但不限于：
  - 场景需求/痛点描述。
  - 实现方案，说明使用了哪些 Agent、Skill、工具组合。
  - 业务价值展示，必须包含 Before/After 数据对比。
  - 复用性说明。
  - 养虾心得（如有）。

决赛阶段：

- 形式要求：需要录制演示视频，最长不超过 8 分钟。
- 内容要求包括但不限于：
  - 场景需求/痛点描述。
  - 实现方案，说明使用了哪些 Agent、Skill、工具组合。
  - 业务价值展示，必须包含 Before/After 数据对比。
  - 复用性说明。
  - 演示完整功能流程。
  - 养虾心得。

官方强调：决赛比拼关键要素不是看写了多少字，而是看跑了多少活、省了多少事。

## 参赛指南

### 这场比赛是什么，不是什么

官方希望参赛者在正式报名之前先建立清晰认知：推荐什么、不推荐什么；什么重要、什么次要。

它和 Skill 大赛的区别：

| 对比维度 | Skill 大赛 | 出钳吧 · 龙虾主理人 |
| --- | --- | --- |
| 定位 | 基座，做宽 | 塔尖，拔高 |
| 考察对象 | Agent 体系中的一个关键部件（Skill） | 整个 Agent 在解决业务问题时的全流程解决方案 |
| 评审重点 | 部件本身的水平、质量、性能以及设计的精妙程度 | 整体效果和落地程度 |
| 核心问题 | 这个工具做得好不好？别人能不能用？ | 解决的是不是一个好问题、大问题？解决效果怎么样？ |
| 门槛 | 低门槛，人人可参与 | 高门槛，面向能拿出真实成果的少数 |

一句话概括：Skill 大赛评的是“能力”，看你做的零件精不精；出钳吧评的是“战果”，看你用 Agent 干出了多大的事。

官方最希望看到：

- 全流程解决方案：不是某一个环节的优化，而是能到端地解决一个完整业务问题。
- 显著的提效跃迁：相比传统方式，有革命性的生产力提升，而非边际改善。
- 好问题、大问题：选择的问题本身是否值得被解决，解决之后能带来多大的价值。

### 入围阶段：方案的合理性与想象空间

入围阶段主要看方案本身“值不值得做”和“有没有可能做成”。

方案的意义与技术契合度：

- 必须是一个有意义的问题：方案要解决一个真实存在的业务痛点，而不是为了参赛而构造的伪需求。
- 技术框架的自然契合：方案应通过 Agent 技术框架更容易地实现，而不是为了使用而硬套。若业务在传统模式下已经运行得非常良好，却硬要转化成 Agent 方式，可能不符合我们的探索初衷。

业务问题的想象空间与提效潜力：

- 人工经验瓶颈：办公场景中存在大量不得不依赖人工经验来处理的环节，这些环节往往是整个流程的效率瓶颈。
- “老师傅手感”的数字化：结合 Agent 能力，将一些带有不确定性判断、依赖经验积累的技术工作转化成 Agent 的思考方法或特定 Skill。

通过这种方式，方案不只是做一个单点提效，而是找到公司全流程现优化和自动化过程中最关键的短板，带来整个公司体系级的效率跃迁。

### 复赛阶段：可落地性与真实价值证明

复赛阶段重点考察项目可落地性。项目不能只停留在概念层面，或只在理论上可行；实际没有被真实地制作出来，构想与现实之间存在很大差距，则不能算合格。

官方需要从方案中直观看到：它是真实可跑通、能解决实际问题的。

真实效率提升：

- 如何证明方案与之前相比有明显效率提升。
- 提升必须是真实发生的，而非理论推演。

数据量化支撑：Before & After。

建议用具体量化数字展示：

- 平均运行了多少次？
- 节省了多少时间？
- 有多少人投入使用？
- 替代了多少工作步骤和重复性问题？
- 规避了多少潜在风险？

迭代打磨的成熟度：

- 复赛阶段希望看到项目真正有意义。
- 通过落地过程，应该已经找到一些“手感”，并得到基于真实业务场景的反馈。
- 方案至少要经过几轮迭代和打磨，使其达到相对成熟、接近产品化的状态。

### 决赛阶段：系统性思维与主动智能

决赛阶段需要筛选优秀团队，并让最具想象力的“养虾人”个体获得更大的展示空间。考量重点除了解决实际问题的程度之外，还主要看以下方面。

技术架构的系统性思维：

- 多系统联动：考察系统之间是否存在积极的联动合理性，以及是否采用了多 Agent 架构。
- 协同联动能力：重点在于 Skill 与其他智能体之间能否产生高效的协同联动能力。能够用一个 Agent 解决问题，与能够带领十个 Agent 同时解决问题，其背后的“养虾能力”有代差。
- 优雅的蓝图：希望看到选手对 Agent 的理解达到较高水平，并能以此勾勒出更优雅、更有想象力、且具备可扩展效应的蓝图。

基于平台特性的主动智能：

- 平台能力的充分利用：考察方案是否在平台特性上找到新的机会，并实现低成本落地。平台具备主动智能、记忆能力、极强的场景上下文理解能力，以及专属于个人或组织的知识库。当前解决方案是否充分利用这些智能性，是衡量“数字员工”与普通工具最大区别的标准。
- 从工具到数字员工：工具只需保质保量完成被动指令，但数字员工必须具备主动智能，能够察觉并意识到关键决策者可能忽略的信息盲区，提供具有情感价值和决策价值的产出。

叙事与产品化：

- 整个方案的叙事系统和选手的口头表达也至关重要。
- 好方案如果无法被很好地展示出来，非常可惜。
- 项目本身的完整度以及过程中的呈现，都是体现优秀解决方案出品人素质的重要因素。
- 决赛会综合复赛入围环节的各项表现，最终优中选优。

## 对 Roboclaws 参赛包装的直接启发

后续讨论 Roboclaws 报名材料时，应优先围绕以下问题展开：

1. 业务痛点是什么：例如 Agent 驱动机器人任务的验证、复盘、评审和复用成本高，行为证据容易散落在日志、截图和临时脚本里。
2. Agent 为什么自然契合：Roboclaws 的核心价值不是单个脚本，而是让 Agent 在任务、Skill、MCP 工具、地图、轨迹、报告和私有评分边界之间形成完整闭环。
3. Before/After 怎么量化：可围绕一次 demo 构建成本、复盘耗时、可见证据完整度、任务复用次数、人工检查步骤、错误定位时间等指标设计。
4. 决赛叙事重点：不要只讲“机器人仿真技术栈”，而要讲“把不可审查的 Agent 机器人行为变成可复盘、可比较、可沉淀的业务资产”。
