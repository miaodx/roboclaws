## §1 执行摘要

**本周期最重要的三条结论：**

1. **三条 §Triggers 已触发，但无一要求路线图重构。**
   - **Substrate major event**：Isaac Lab 3.0 Beta 落地（ground-up 架构重写、Newton/PhysX 多 backend、kit-less 安装）。
   - **Model-generation jump with vision impact**：GPT-5.5 公开（2026-04-23），叠加 Opus 4.7 早 9 天落地的视觉跃迁。
   - **Peer/upstream harness change**：Anthropic Managed Agents 在四周内堆栈式落了 memory（公测）、dreaming（research preview）、multiagent orchestration（公测）、outcomes（公测）。
   
   三者都是"在已知方向上加速"，不是"颠覆已下注方向"。四模式架构维持，H1–H8 全部维持或加强。

2. **H8（多 coding-agent + 多机器人 = 前沿）仍然成立，但窗口在从两个方向同时收缩。** 窗口内出现的最近邻：
   - **CooperBench**（arXiv:2601.13295，2026-01）在代码侧给出**量化反证**——多 coding agent 协作的成功率比各自单干平均低 30%。这是 H8 的代码侧轴第一次有了具名 counter-evidence。
   - **LLM-Foraging**（arXiv:2605.01461，2026-05-02）在 robot 侧合龙了"多 LLM 决策器 + 多模拟机器人"，但 LLM 出的是离散战术选择，不是 code-as-action。
   - **CaP-X**（arXiv:2603.22435，2026-03）合龙了"coding agent + 模拟机器人"，但单 agent 单 robot。
   
   两个轴都接近但还没在同一份公开工作里合龙。H8 仍真，但**应升级为重点观察**。

3. **H2（双 substrate 下注）在本期获得最强正面证据：MolmoSpaces 在 4 周内零多智能体信号。** 系统性扫描其 commit / issue / PR / release / Allen AI blog 全部维度，与"multi-agent / multi-robot / swarm"语义相关的工件**完全为零**。这不是"还没决定"——这是 Allen AI 路线在本周期内**没朝那个方向移动一步**。AI2-THOR 仍是多智能体的唯一可走 substrate，与 vectors 卡片当前下注一致。

## §2 强制 Delta

vectors 卡片 §Status 节点名要求本期可见的四项 delta。

### 2.1 OpenClaw governance

**2026-04-29 "rough week"** 事件：OpenClaw 在 4/24–4/29 期间多个版本因插件依赖修复死循环、bundled/external 插件半拆分、metadata 不稳定，导致 gateway 在多个消息平台行为退化、用户大规模降版。创始人 Peter Steinberger 在 [openclaw.ai/blog/openclaw-rough-week](https://openclaw.ai/blog/openclaw-rough-week) 公开致歉，关键判断："This was not one bug...OpenClaw was still too founder-driven. Too much release, review, packaging and support work sat with me."

**结构性回应**：宣告即将通过 OpenClaw Foundation + OpenAI sponsorship 组建专职团队，并在 5 月晚些时候宣布 LTS 发布线（与快速通道并行）。**这是 2026-Q1 基金会移交后第一次出现创始人显式承认必须分担的信号**——治理姿态向 Linux Foundation 项目典型形态靠拢。

**NemoClaw 整合**：NVIDIA 在 GTC 2026（2026-03-16）发布 NemoClaw 早期预览，以 NVIDIA OpenShell 沙箱（Landlock + seccomp + netns）+ Privacy Router + Nemotron 本地模型 wrap OpenClaw。Apache 2.0，本窗口期内尚无 GA。Cobus Greyling [cobusgreyling.substack.com/p/nvidia-nemoclaw](https://cobusgreyling.substack.com/p/nvidia-nemoclaw) 将其定性为 "platform-capture move" 而非纯技术突破，与 NVIDIA 官方公告 [nvidianews.nvidia.com](https://investor.nvidia.com/news/press-release-details/2026/NVIDIA-Announces-NemoClaw-for-the-OpenClaw-Community/default.aspx) 双源交叉确认。

**对 H4 的意义**：OpenClaw 治理在"基金会 + 大厂 enterprise wrap + LTS 分支"三条线收敛。Mode 2 的底盘没有变，但稳定性与 audit 路径更清晰。**这是 H4 在本周期最强的支撑信号**。

### 2.2 MolmoSpaces multi-agent

**窗口内零信号**——这是本期最有说服力的"无变化"。系统性扫描 4 个维度：

- **Repo 工件**：开放 issue / PR / commit / release 在"multi-agent / multi-robot / swarm"语义上全部零命中。仓库内文本搜索亦零命中（"multi-agent"仅在隔壁 ai2thor 的旧 issue 中出现）。
- **Release 公告**：v0.1.0（2026-05-05）的 release notes 没有任何多智能体表述。
- **Allen AI blog**：5 月内**没有** MolmoSpaces 后续文章；PRIOR 系列最新仍是 2026-03-24 的 MolmoBot-Datagen（单机器人 zero-shot）。
- **官方文档措辞**：README 仍局限"single-arm and dual-arm systems"——这里 dual-arm 是双臂单机器人 bimanual，不是多机器人协调。来源：[allenai.org/blog/molmospaces](https://allenai.org/blog/molmospaces)。

**对 H2 的意义**：vectors 卡片当前的下注（"AI2-THOR 拿 navigation/territory + MolmoSpaces 拿 manipulation/cleanup，多智能体仍只能依赖 AI2-THOR"）在本周期**获得强证据支撑**。MolmoSpaces 路线在多智能体方向上不光是"还没做"，而是**4 周内没朝那个方向移动一步**。

但这同时意味着：依赖 manipulation 多智能体的扩展路线**短期内仍被上游卡住**，roboclaws 自身要么在 AI2-THOR 范围内做多智能体，要么等更长。

### 2.3 多 coding-agent + 多机器人公开前案（H8 关键）

窗口内仍未出现严格匹配 H8 的公开工作（"多 code-as-action coding agent 同时驱动多个模拟机器人在共享任务上"）。**但两个轴各自的最近邻在本窗口都已落地：**

**代码侧轴 — CooperBench**（arXiv:2601.13295，Stanford 等，2026-01-19/26）：652 任务两 agent 协作编码 benchmark（Python/TypeScript/Go/Rust 12 个开源库）。Abstract 量化结论："agents achieve on average 30% lower success rates when working together compared to performing both tasks individually, across the full spectrum of task difficulties." [arxiv.org/abs/2601.13295](https://arxiv.org/abs/2601.13295)

这是 H8 在代码侧第一次有了 named counter-evidence——多 coding agent 协作本身的失败模式已被量化。任何 H8 路线设计都必须吸收这一点。

**机器人侧轴 — LLM-Foraging**（arXiv:2605.01461，2026-05-02）：多 LLM 决策器在 CPFA 状态机的三个 structured decision points 上为分散群体机器人觅食做战术决策。**关键限制**：LLM 出的是离散战术 waypoint 选择，**不是 code-as-action**。[arxiv.org/html/2605.01461v1](https://arxiv.org/html/2605.01461v1)

**单轴合龙 — CaP-X**（arXiv:2603.22435，2026-03-23）："agents control robots by synthesizing and executing programs that compose perception and control primitives." 单 agent + 单机器人，但精确匹配了 H8 中"coding agent 驱动 simulated robot"那一侧。[arxiv.org/abs/2603.22435](https://arxiv.org/abs/2603.22435)

**AAMAS 2026** 在 2026-05-25 至 29 召开（塞浦路斯 Paphos）。本检查点 as-of 2026-05-22，**proceedings 尚未公布**——5/29 后必须 sweep 一遍论文集，是否有合龙工作出现。这是 §6 的硬约束 carry-forward。

**对 H8 的意义**：frontier 仍真，但窗口正从两侧同时收缩。代码侧已经有公开的"做不好"证据（CooperBench 30% 下降），机器人侧已经有公开的"靠 LLM 多决策器"路线（LLM-Foraging）。要么有团队**有意识地**做两轴合龙，要么 H8 frontier 状态可能在 2–3 个月内被关闭——而不是更长。

### 2.4 真机桥接（H5 / H8 的真实机器人延伸）

roboclaws 自身仍在向真实机器人方向推进。**本检查点不下沉到具体 PR / 分支状态**（实现层在 PR 周期内会变）。架构层上的判断：

- **VLA 候选选型**仍维持 2026-04 决定的"等 substrate 落定再选"。Isaac Lab-Arena 把 GR00T 与 OpenPi 拉到同一 evaluation pipe，但还在 alpha；vectors 卡片 V7 watchlist 不需要改变。
- **LeRobot 生态**：LeRobotDataset v3 schema 在窗口期推进，但 v3 与 v2.1 在数据合约层面兼容（差异在物理打包结构）。这对 roboclaws 是"未来再说"的兼容性信号，不是当前的架构压力。
- **机器人 MCP peer 生态**：isaacsim-mcp / ros-mcp-server / NVIDIA Isaac MCP 等多个仓库本窗口持续维护，但**没有一个**成为 dominant 标准。这与 H3（MCP + SKILL.md 是当前最佳打包）一致——MCP 协议层稳定，server 实现层百花齐放，roboclaws 自家做自家的不踩坑。

## §3 七个 Vector 增量更新

### V1 — Harness 上游 + driver SDK

**Anthropic Managed Agents 四件套（2026-04-23 起到 2026-05-06 全部公测/preview）：**

- **Memory**（2026-04-23 公测）：agent 把记忆作为 filesystem 文件，开发者可在 Console / API 查看编辑。来源：[claude.com/blog/claude-managed-agents-memory](https://claude.com/blog/claude-managed-agents-memory)。Rakuten 自报数据："97% fewer first-pass errors at 27% lower cost and 34% lower latency"（同来源）。
- **Dreaming**（2026-05-06 research preview）：session 间运行的后台 review 过程，提取 pattern、curate memory。可选 auto-update 或 review-before-land。
- **Multiagent orchestration**（2026-05-06 公测）："up to 20 parallel specialists" 在一个 session 内协同。
- **Outcomes**（2026-05-06 公测）：rubric 驱动的 grader 架构，独立 Claude 实例评分。

四件套加上同期落地的 self-hosted sandboxes（公测）+ MCP tunnels（research preview），意味着 **harness-as-service 已经把"多 agent 编排 + 持久记忆 + 自动评分 + 客户机房执行 + 私网工具"打成一个集成包**。

**对 H4 + H8 的意义**：Mode 3（harness-as-service）的底层基础设施在 4 周内集成度跳了一档。H8 路径的实现成本**显著下降**——做"多 coding agent 同时驱动多机器人"不再需要自己写 orchestrator，可以直接挂在 Managed Agents 多 agent SDK 上。这是 vectors 卡片应新增一条 §Trigger 的方向（详 §5）。

**Cursor 多 coding-agent 上限**：[cursor.com/blog/self-driving-codebases](https://cursor.com/blog/self-driving-codebases) 自报代码侧 production 峰值——"~1,000 commits per hour across 10M tool calls over a period of one week"。这是当前公开可参考的"多 coding agent 同时在一个代码库"production-grade 上限。**embodiment 端没有对应数字**。

**反例信号**（未来）：某个 provider 在 robotics 相关任务上显著吞掉其他。

### H8 — 多 coding-agent + 多机器人 = 无 precedent 的前沿

**结论：维持，升级为重点观察。** 两个轴的最近邻在本窗口都到位——CooperBench 给出代码侧的量化反证（多 agent 协作平均下降 30%），LLM-Foraging 合龙了多 LLM + 多 robot 但非 code-as-action，CaP-X 合龙了 coding agent + simulated robot 但单 agent。任意一个团队**有意识地**做两轴合龙都可能让前沿 status 在 2–3 个月内被关闭。

CooperBench 的 30% 量化反证尤其重要：H8 的多 agent 路线设计**必须吸收**这一发现——多 coding agent 协作本身的失败模式已被量化，roboclaws 在机器人侧扩展时不能假设这个问题不存在。

**反例信号**（未来）：任何描述 code-as-action coding agent 同时驱动多 simulated robot 的公开工作。
