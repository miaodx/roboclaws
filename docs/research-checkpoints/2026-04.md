## 0. 关于本文档

### 0.1 为什么要做这个 checkpoint

`roboclaws` 早期做选型时，我们在很短的时间内确定了"直接调 VLM API + OpenClaw Gateway + code agent"这三层栈。当时这个判断是合理的——在那个时间点上，能让多个 VLM 实例稳定驱动 AI2-THOR 仿真机器人的生产级方案就这么几个。

但是 agent 框架生态在 2025–2026 这一年是爆炸式增长的：

- 仅 OpenClaw 一个项目就衍生出 40+ 个 100+ stars 的变体，覆盖 8 种语言和 4 种部署形态；
- agent SDK 层（Pi、Claude Agent SDK、OpenAI Agents SDK、Smolagents 等）从无到有再到收敛；
- 机器人 VLA 栈（NVIDIA GR00T、Physical Intelligence π 系列、LeRobot）开始成为可即插即用的基础设施；
- MCP / A2A / ACP 三个协议从草案变成了产业事实标准。

在这种节奏下，**任何"选型决定"都有保鲜期**。我们当初挑 OpenClaw 是合理的，但**今天如果重新选**，候选集会完全不同——Nanobot、Hermes Agent、Smolagents、Voyager 风格的自建栈，每一个都可能是更好的起点。

这个文档要做两件事：

1. **冻结当前时间点的认知**：把 2026 年 4 月这一刻我们对生态的理解、对 roboclaws 取舍的判断完整记录下来，包括*哪些是基于证据的结论*、*哪些是经验直觉*、*哪些是未来需要验证的假设*。
2. **建立可持续更新的机制**：让"每月调研一次"成为一种轻量、可重复、可验证的习惯，而不是每隔半年发现自己又落后了再做一次大调研。

文档不是结论，是**调研的快照**。每个 checkpoint 都应该可以独立读懂，但放在一起又能看出生态的演化轨迹。

### 0.2 每月更新方法

每月更新一次，节奏定在月底（每月最后一周）。每次产出一个新的 `roboclaws-checkpoint-YYYY-MM.md` 文件，与历史 checkpoint 并列保存，**不覆盖**——这样可以追溯任何一个判断是在什么背景下做出的。

每次更新跑下面这套流程：

**步骤 1 — 增量调研（耗时约 1–2 小时人工 + 30 分钟 agent 时间）**

跑一组针对性的 deep research 查询，覆盖 5 个固定方向：

1. **OpenClaw 严格家族增量**：与上个 checkpoint 比，有哪些新出现的 100+ stars 变体？哪些消失了或停止维护？
2. **Agent SDK 层动态**：Claude Agent SDK / OpenAI Agents SDK / Pi / Smolagents / LangGraph 这五家这个月有什么 release 或方向变化？
3. **VLA / 机器人栈动态**：GR00T、openpi、LeRobot、Isaac Lab 有什么新版本？有没有新出现的开源 VLA？
4. **协议层**：MCP、A2A、ACP 这个月有没有规范变更或重要的新 server？
5. **学术前沿**：arXiv 上"LLM agent survey"、"VLA survey"、"multi-agent embodied"过去 30 天有什么值得读的新综述？

**步骤 2 — 增量整理（耗时约 1 小时）**

把调研结果对照上个 checkpoint，挑出 3 类需要更新的内容：

- **新增项目**：加到 §3.2 / §3.3 表格，标注 "(新增于 YYYY-MM)"。
- **状态变化**：原项目 stars 大幅变化（>30%）、改名、停止维护、被收购、关键人物离开——在原条目上加一个 `[YYYY-MM 更新]` 标注。
- **判断修正**：如果新证据让某个 §6 建议不再成立，**不要删除原建议**，而是在下面加一段 "[YYYY-MM 修正]" 说明为什么改变。这样后续读 checkpoint 的人能看出"我们是怎么改变想法的"。

**步骤 3 — 写新 checkpoint（耗时约 1–2 小时）**

复制上个 checkpoint 作为模板，更新：

- 文件头的日期、下次更新日期；
- §1 执行摘要——重写，反映本月最重要的 3–5 个变化；
- §3 / §4 / §5 / §6 各章节——按步骤 2 的标注更新；
- §7 公开问题——勾掉本月已回答的，加上本月新发现的；
- §8 变更日志——新增一节 `## YYYY-MM-DD` 列出本次的所有变更。

**步骤 4 — Review 与 commit**

人工 review 整篇文档。重点检查：

- 有没有过度采信单一信源（特别是 SEO 内容站的判断）？
- 有没有"这个月没新东西所以照抄上月"的偷懒？
- §6 的建议有没有跟 §3–§5 的证据脱节？

review 通过后 commit 到 repo 的 `docs/research-checkpoints/` 目录。

**触发临时调研的条件（不等月底）：**

- 出现"模型代际跃迁"——新一代旗舰模型（如 Claude Opus 5、GPT-6、GR00T N2）发布；
- roboclaws 自身做出重大架构决策需要现场调研支撑；
- 某个我们已采纳的项目（如 Hermes Agent）发生重大变更（停维护、license 变更、安全事件）；
- 学术上出现 paradigm-shifting 的新工作（罕见，过去一年大概 2–3 次）。

### 0.3 信源质量与可信度声明

调研中遇到的信源大致分四档，引用时心里要分清：

**A 档（一手、可信）**：项目官方 GitHub repo、官方 blog（Anthropic、OpenAI、NVIDIA Developer、HuggingFace）、arXiv 论文、Linux Foundation 官方公告、维护者本人的技术博客。这些可以直接引用，但仍需验证发布时间——arXiv preprint 不等于已发表，官方 blog 可能反映的是公关角度。

**B 档（二手、需交叉验证）**：技术媒体（The Information、VentureBeat、TechCrunch）、独立工程师的对比博文、HackerNews/Reddit 讨论的高赞回答、产业咨询报告。这些通常方向对，但具体数字和时间线要核对一手信源。

**C 档（聚合站、谨慎使用）**：clawclones.com、shelldex.com、awesome-claws 这类聚合/对比门户。它们对**生态广度**的反映有用（"哦，原来还有这个项目"），但对**任何具体定量声明**（stars 数、安全 issue 数、市占率）都要按 ±20% 看待，且最好回到一手 repo 验证。

**D 档（SEO / 内容营销，多数不引用）**：vellum.ai、qubittool.com、composio.dev、scriptbyai.com、aimagicx.com 这类站发布的"2026 年最佳 X 替代品"导购文章。它们的**结构化结论**（某种工具适合某种场景）有时有参考价值，但**具体声明**（性能数字、市场判断）几乎都是为 SEO 服务，**绝大多数情况下应该忽略**。如果某个判断只在 D 档信源里出现而无 A/B 档佐证，请在文档里标注 "[未充分验证]"。

**特殊提示**：本调研的执行时间是 2026 年 4 月，但很多被引用的文章发布日期标的是"2026"——这在中文/西方常规历法下显得"未来日期"。这不是错误，是 SEO 内容站常用的"提前年份化"策略。判断时以**实际发布时间戳**为准，不被标题里的年份带偏。

### 0.4 阅读指南

文档比较长，按你的目的选择切入点：

- **想快速了解结论**：读 §1 执行摘要 + §6 建议（约 10 分钟）。
- **想理解 OpenClaw 生态全貌**：读 §3（约 25 分钟，含表格扫读）。
- **想思考架构怎么切**：读 §4（约 30 分钟，4 个视角各 7 分钟）。
- **想做选型决策**：读 §6 + §7（约 15 分钟）。
- **想跟进学术前沿**：读 §5.1 + §5.3（约 15 分钟）。
- **想做下一次 checkpoint**：读 §0.2 + §8 变更日志 + §7 公开问题（约 10 分钟），知道这次该补什么。

文档里使用的几个约定：

- **加粗**用于强调关键判断或项目名首次出现。
- *斜体*用于表示"产业内的术语"或"需要被定义的概念"。
- 反引号 `code` 用于文件名、API 名、命令行工具。
- 中英混排：技术术语保留英文（agent、harness、skill、SOUL、persona、channel、sandbox 等），它们在中文社区已是默认。
- "L-Hi / L-Mid / L-Lo / L-Sim" 是本文档自创的具身栈层级标签，仅在 §4.4 之后使用。
- "L1 / L2 / L3" 是你最初提出的抽象层级标签，仅在 §4.1 使用。两者不冲突——它们是同一问题的两个不同视角。

## 1. 执行摘要

**生态层面的 4 个判断：**

1. OpenClaw 生态比预期大得多——存在 **44+ 个严格家族变体**（fork、重写、inspired-by），覆盖 8 种语言和 4 种部署形态。但其中真正与 roboclaws 高相关的只有 **5–6 个**（Hermes Agent、Nanobot、CoPaw/QwenPaw、TinyClaw、KafClaw、ClawTeam-OpenClaw）。"还有 30 多个变体"这件事的主要价值是确认了**模式收敛**——它们做的事高度相似，差异在运维特性而非架构思想。
2. 同期"agent 框架格局"已经收敛到一个 **5–6 层的产业事实栈**：Model → Protocol（MCP/A2A/ACP）→ Tools → Sandbox（E2B/Modal/Daytona）→ Harness（Claude Agent SDK / OpenAI Agents SDK / Pi / Smolagents / LangGraph）→ UI/channel。Memory 跨层、Multi-agent 编排正交。这意味着新项目不再需要"自己造一切"。
3. 机器人专属栈也收敛到 **双系统/三系统层级**：System-2 高层 reasoner（VLM）→ System-1 中层 VLA action expert（GR00T System-1 / π₀ / OpenVLA）→ 低层全身/电机控制器（Isaac Lab RL 训练的 whole-body controller）。这个收敛在 2025 年下半年到 2026 年 Q1 期间发生。
4. **SOUL.md / IDENTITY.md / USER.md / MEMORY.md 已成为 per-agent persona 的事实标准**——OpenClaw、Nanobot、Hermes、NanoClaw、OpenPersona、soul.md、SoulCraft 等 7+ 项目都消费或生产这套约定。Anthropic Claude 内部"Soul Document"泄露事件（2025 年 11 月）也间接验证了这个概念在前沿实验室里的存在。**roboclaws 应该直接采用**。

**对 roboclaws 的 5 个具体建议：**

1. **不要继续基于 OpenClaw（TypeScript）扩展**。短期换成 Python 栈下的 Hermes Agent 或 Nanobot 作为 L3-gateway 角色，长期参考 Voyager + Generative Agents 的"自建小型 Python harness"路线。
2. **从第一天就用 SKILL.md / SOUL.md / MEMORY.md 约定**——这套是免费的可移植性，未来无论换哪个 harness 都还能用。
3. **MCP 化所有外部工具**——AI2-THOR `controller.step`、Isaac Lab/ROS 接口都包成 MCP server。这给你后续接入任何 harness 的零成本通道。
4. **架构文档同时给 4 个视角**：抽象层级（L1/L2/L3，对外解释好用）、职责（perception/memory/…/action，做组件 ablation 好用）、部署形态（CLI/gateway/cloud/embedded，做运维决策好用）、具身栈（reasoner/skill/controller/sim，做 Phase-3 迁移规划好用）。它们不冲突，是同一问题的不同切片。
5. **Phase-3 准备工作现在就启动**：从今天起，AI2-THOR rollout 数据用 LeRobotDataset 格式存。这样未来无缝迁移到 Isaac Lab + GR00T / openpi。

**3 个我们仍不确定、需要下次 checkpoint 验证的问题：**

- code-as-action（Smolagents 风格）vs JSON tool-call（OpenAI 风格）在仿真离散导航上谁更稳？尚无公开 benchmark。
- Hermes Agent 自动 skill 创建能力在多 agent 仿真场景里是否真的可用？还是只在单 agent 个人助理场景测过？
- Phase-3 做 G1 humanoid 时，应该选 NVIDIA GR00T N1.7 还是 Physical Intelligence π₀.₅？两者授权模式不同，需要做一次专门评估。

## 2. 背景：roboclaws 现状与约束

为了让本文档读起来不依赖外部上下文，简要陈述 `roboclaws` 当前的状态和已知约束。下次 checkpoint 时如果这些发生变化，请先更新本节。

**项目定位**：多个 VLM/agent 实例在 [AI2-THOR](https://ai2thor.allenai.org/) 仿真环境中控制机器人，进行 territory control（领地控制）和 cooperative coverage（协同覆盖）类任务。本质上是一个**多 agent 具身 AI demo / 研究平台**，不是个人助手、也不是生产服务。

**当前栈（2026 年 4 月）**：

- **L1 — 直接调 VLM API**：Claude / GPT-4o / Kimi 通过原生 SDK 调用，自己解析 tool call，自己写 retry 和错误处理。
- **L2 — OpenClaw Gateway + Skills**：让 OpenClaw 作为 long-running gateway，每个 robot 一个 session，skill 写 AI2-THOR 控制逻辑。
- **L3 — Code agent**：让 agent 自己写 Python 代码段控制机器人（Smolagents 风格），仿真器作为代码执行的 sandbox。

三层是并行的实验线，不是从底向上的依赖关系——这点和"普通 agent 应用栈"不同，需要在写文档时讲清楚。

**仓库构成**：90.8% Python，少量 shell 和配置。OpenClaw Gateway 调用是通过子进程或 RPC 调外部 TypeScript 进程实现的——这是当前的一个"语言阻抗"痛点。

**Roadmap**：

- **Phase 1**（已完成）：单 agent 在 AI2-THOR 中按指令完成离散导航任务（MoveAhead/RotateLeft/…）。
- **Phase 2**（进行中）：多 agent，引入 territory control 和 cooperative coverage，per-agent SOUL/personality，episodic memory。
- **Phase 3**（规划中）：迁移到 NVIDIA Isaac Lab + Unitree G1 humanoid + RL locomotion policy。这是从"VLM 调离散动作 API"跨到"VLM + VLA action expert + 全身控制器三段式栈"的质变。

**已知技术约束**：

- 优先 Python 栈（90.8% 代码、研究员和 collaborator 的母语）。
- 必须支持多 LLM provider 切换（实验需要对比 Claude vs GPT vs Kimi）。
- 每个 agent 需要独立的 SOUL/personality + 独立的 episodic memory，不能共享。
- 需要支持 agent 间通信（territory negotiation、cooperative coverage handoff）。
- Phase-3 时要能跑 Isaac Lab 训练循环（GPU 并行 sim），所以底层 sandbox/runtime 必须 GPU-aware。

这些约束驱动了下面所有判断。下次 checkpoint 时如果约束变了（比如换主语言、加新 provider、改 Phase 计划），先更新本节再写后面。

## 3. OpenClaw 生态全景

### 3.1 OpenClaw 与 Pi/pi-mono 的关系（一个常见误解的修正）

调研早期我们犯过一个错误：把"Pi SDK"理解为 [Physical Intelligence](https://www.physicalintelligence.company/) 公司的 π₀ 机器人模型 SDK（`openpi`）。这是**完全不同的两个东西**，但因为它们都常被简称为"Pi"，初次接触很容易混淆。把这个差异写在前面，避免下次 checkpoint 又踩同一个坑。

**Pi 系列 1：pi-mono（Mario Zechner 的 agent SDK）**

- 维护者：[Mario Zechner](https://twitter.com/badlogicgames)（GitHub 上是 `@badlogic` / `@badlogicgames`），libGDX 游戏引擎的作者。
- 仓库：[`badlogic/pi-mono`](https://github.com/badlogic/pi-mono)
- 性质：TypeScript monorepo，agent SDK 工具包。
- 设计理念：不满 Claude Code 越来越复杂，反着做了一个"4 个工具、系统提示词不到 1000 token"的极简内核。理念是"留下什么比加上什么更重要"。
- 子包：
  - `pi-ai`：跨 provider 的 LLM 通信层（Anthropic / OpenAI / 自有部署）；
  - `pi-agent-core`：带 tool calling 的 agent loop 内核；
  - `pi-coding-agent`：完整的 coding agent CLI（带 read/edit/write/bash 工具、session 持久化）；
  - `pi-tui`：CLI 终端 UI 库；
  - `pi-pods` / `pi-mom`：vLLM pod 和 Slack bot 等周边。
- 运行模式：interactive（交互式）/ print（一次性 JSON 输出）/ RPC（进程集成、JSONL framing）/ SDK（嵌入到自己的 app）。
- 在生态中的位置：**OpenClaw 的内核**。OpenClaw README 里有一句 "Special thanks to Mario Zechner for his support and for pi-mono"，证实了这一点。Anthropic 的 Claude Agent SDK 可以看作 Pi 的 Python 同类，OpenAI Agents SDK 也是同一个抽象层级的产物。

**Pi 系列 2：openpi（Physical Intelligence 的机器人 VLA）**

- 维护者：[Physical Intelligence](https://www.physicalintelligence.company/)，由 Sergey Levine、Karol Hausman 等创办的机器人基础模型公司。
- 仓库：[`Physical-Intelligence/openpi`](https://github.com/Physical-Intelligence/openpi)
- 性质：Python（JAX + PyTorch），机器人 Vision-Language-Action 基础模型。
- 内容：π₀ / π₀-FAST / π₀.₅ 三代模型的开源实现，能跨双臂、单臂、移动机器人输出连续电机指令；权重和代码都开源；支持 LeRobot 数据集格式；通过 websocket 做远程推理。
- 在生态中的位置：**机器人 VLA 栈的底层 action expert**，对标 NVIDIA GR00T 的 System-1。和 agent SDK 完全是不同抽象层。

**两者的关系**：没有关系。它们恰好都被简称为 "Pi"，恰好在 agent / robotics 这个交叉领域被我们这种项目同时关注，恰好都是开源的——但维护者、技术栈、目标场景、抽象层级全都不同。

**正确的"Pi"理解方式**：

- 当讨论 **agent SDK / harness / OpenClaw 内核**时，"Pi" = pi-mono。
- 当讨论 **机器人 VLA / action 模型 / Phase-3 栈**时，"Pi" = openpi / π₀。
- 在本文档里，凡提及前者写 "Pi (pi-mono)"，凡提及后者写 "openpi" 或 "π₀"。

**这个混淆为什么值得写出来**：因为下次 checkpoint 时如果新读者只读结论不读对话历史，很容易再犯一次。而这两个 Pi **都**是 roboclaws 的关键依赖（一个是当前 agent 层的内核选项，另一个是 Phase-3 的候选 VLA），混淆它们会导致严重的架构判断失误。

### 3.4 从变体格局浮现的 9 个稳定模式

70+ 项目里浮现出几个稳定模式。这些模式比"哪个变体好"更重要——它们告诉你**生态在收敛到什么形状**，从而预测下次 checkpoint 时哪些会变、哪些不会变。

**(a) "Agent harness" 现在是产品，不是模型。** 每个变体——包括 Rust/Zig/Go 重写——都几乎一模一样地保留了 agent loop、工具表面（read/write/edit/bash + skill）、SOUL/persona 约定。竞争点是*运维特性*（内存占用、隔离模型、部署形态、skill 市场广度）。Anthropic 自己 2026 年 4 月 Claude Agent SDK 的"显式把 harness 与 sandbox 分开"转向、以及公开的 CORE benchmark 结果（同一个 Opus-4.5 在 Claude Code harness 下得 78%，在 Smolagents 下 42%）都在强化一个结论：***harness 是模型的乘数*，不是薄包装**。

**意涵**：roboclaws 当前用 OpenClaw 作为 harness 是合理选择，但当我们自己写 harness 时（哪怕只是一层薄的 Python wrapper），需要认真对待——它对最终性能的影响可能比换 LLM provider 还大。

**(b) SOUL.md 正在成为事实上的 persona 标准。** 至少 7 个独立项目（OpenClaw 核心、NanoClaw、Hermes Agent、soul.md、SoulCraft、OpenPersona、opensoulmd CLI）消费或生产 SOUL.md / IDENTITY.md / USER.md / MEMORY.md。ZeroClaw 引入 AIEOS（同一思路的 JSON 方言）作为可移植规范。Anthropic Claude "Soul Document" 泄露事件（2025 年 11 月）验证了这个概念在前沿实验室内部的存在。

**意涵**：对 roboclaws 来说这是**已解决的领域**——直接采用即可。每个仿真机器人一份 SOUL.md（性格、策略偏好、风险态度），加一份 MEMORY.md（episodic 反思）。

**(c) 安全模型是主要差异化。** 变体清晰地分成 4 档：(i) 共享进程信任（OpenClaw、Pi、Smolagents 默认）；(ii) 容器/microVM 隔离（NanoClaw、Moltis、E2B、Modal）；(iii) WASM/能力沙箱（IronClaw、OpenFang）；(iv) TEE / 内核沙箱（NEAR AI Cloud、NVIDIA OpenShell 加持的 NemoClaw）。

**意涵**：roboclaws 不需要 TEE，但**需要 container-per-agent**——这样某个流氓 VLM（生成乱七八糟 Python 代码）不会污染共享 sim 或别的 agent 的 memory。E2B 或 Modal 能很自然提供这个隔离。

**(d) 多 channel 广度大体是噪声。** 多数变体竞争"现在我们支持 30+ 聊天 channel"。对机器人仿真 use case，*只有一个* channel 重要：承载 sim observation 和 action 的传输层。

**意涵**：评估变体时**完全忽略 channel 数量**。一个支持 3 个 channel 但设计干净的项目，比一个支持 30 个 channel 但耦合的项目对 roboclaws 价值高得多。

**(e) Memory 架构收敛到 4 层模型**：

```
SOUL/IDENTITY (不可变 persona)
    ↓
MEMORY.md (慢、反思式、agent 自整理)
    ↓
recall/session FTS (快、全文)
    ↓
archival vector (跨 session、语义)
```

Hermes Agent、Letta、OpenClaw、Mem0、Zep/Graphiti 都是这个模式的变体。

**意涵**：实现成本不高——SQLite + FTS5 + 一个小 embedder 就够了，不需要专门的向量数据库。如果想偷懒，直接用 [Mem0](https://github.com/mem0ai/mem0) 或 [Letta](https://github.com/letta-ai/letta) 做这一层。

**(f) L2/L3 边界很模糊**——这点直接影响你的 L1/L2/L3 分类法。"Agent SDK 内核"项目（pi-mono、Smolagents、Claude Agent SDK、OpenAI Agents SDK）和"完整 assistant"（OpenClaw、Hermes、Nanobot、OpenFang）越来越像同一个东西戴着不同的帽子。

经验上的判定信号：***谁拥有 loop***。

- 如果你的代码 `agent.run(...)` 调进去，控制权是你的——你拿的是 SDK / harness（L2）。
- 如果一个长生命周期的 gateway 进程通过 webhook/skill 调进你的代码，控制权是它的——你拿的是 assistant（L3）。

**意涵**：roboclaws 是 SDK use case——仿真器驱动时间，你的代码持有 loop，agent 的 `step()` 嵌到仿真 step 里。这意味着 OpenClaw（L3 形态）天然不太对路，**应该向 L2 形态的 harness 靠**（Hermes 或 Smolagents 或自己写）。

**(g) Skill-as-markdown 是占主导地位的扩展格式。** SKILL.md frontmatter + body 被 OpenClaw、Hermes、Nanobot、Claude Code、Cursor、Codex、Cline、Roo、Goose、OpenCode、AionUi、Letta、SoulCraft、soul.md 都支持。

**意涵**：从第一天就用这个格式写 roboclaws skill（如 `patrol_grid.md`、`negotiate_handoff.md`、`report_territory.md`）。即使现在只是个备忘格式，未来对接任何 harness 都已经准备好。

**(h) 对具身 agent 来说，code-as-action 击败 JSON tool-call。** Smolagents 公开的证据（少 30% LLM step；CodeAgent 用 ToolCallingAgent 时能匹配闭源大模型）、Voyager 的整篇论点（skill 即可执行代码）——都指向同一个方向。

**意涵**：对 AI2-THOR 中的离散导航动作，agent 应该输出 Python 代码（如 `for _ in range(3): controller.step(action="MoveAhead")`），而不是 JSON tool call（`{"tool": "move", "args": {"direction": "ahead", "n": 3}}`）。但**这点尚未在我们的具体场景里 benchmark 过**——见 §7 公开问题。

**(i) 最相关的相邻设计是 Voyager + Generative Agents，不是 OpenClaw。** OpenClaw 是为"几秒内回聊天，持久几年"优化的；Voyager 和 Generative Agents 是为"在仿真器里走几千步，持久 skill library 和 memory stream"优化的。

**意涵**：当 roboclaws 文档里需要给读者一个**架构参考**时，引用 Voyager 和 Generative Agents 比引用 OpenClaw 更准确。OpenClaw 的价值在于它沉淀的**约定**（SOUL/MEMORY/SKILL.md 格式），而不是它的 architecture。

## 4. 分类法：4 个视角看同一个问题

本节是文档的核心拓展。前一版报告把"分类法 C（具身栈）"作为推荐主轴、把其它视角降为附录——但你点出来说**这几个视角应该并列**，每个都从不同角度讲述同一个问题。这是对的。

下面 4 个视角各自独立成章，每个都给：

- **定义**：这个视角怎么切？
- **应用到 roboclaws**：现在的栈在这个视角下长什么样？
- **它最适合回答的问题**：什么时候用这个视角？
- **它的盲区**：什么时候不要用这个视角？

§4.5 给一个综合表格，并讨论 4 个视角怎么组合使用。

---

### 4.1 视角 I：抽象层级（你的 L1/L2/L3）

**定义**：按"距离 LLM 原始 API 多远"切层。

- **L1 — Raw LLM API**：直接 HTTP 或 SDK 调 `messages.create`，自己解析 tool call、自己写 retry、自己拼 prompt。代表：Anthropic SDK、OpenAI SDK、google-generativeai 等。
- **L2 — Agent SDK / Harness**：提供 agent loop（while-tool-call-loop）、tool 抽象、session 管理，但不绑定 channel/UI。控制权在你手里，你的代码 `agent.run()` 调进去。代表：Pi (pi-mono)、Claude Agent SDK、OpenAI Agents SDK、Smolagents、LangGraph。
- **L3 — 完整 Assistant Platform**：长生命周期进程，自带 channel（聊天/邮件/Slack/…）、skill marketplace、memory 持久化、cron/heartbeat、persona 系统。控制权在它手里，它通过 webhook/skill 调进你的代码。代表：OpenClaw、Hermes Agent、Nanobot、AstrBot、CoPaw/QwenPaw。

**应用到 roboclaws**：

| 当前实验线 | 层级 | 落点 |
|------------|------|------|
| 直接调 VLM API（Claude/GPT-4o/Kimi） | L1 | Anthropic SDK + 自写循环 |
| OpenClaw Gateway + Skills | L3 | OpenClaw 进程 + AI2-THOR control skill |
| Code agent | L2 | 类 Smolagents 模式，自写 Python 沙箱 |

**它最适合回答的问题**：

- "我应该用哪个抽象层做 X？"——根据需要的控制粒度选。
- "这个项目和那个项目处于同一层吗？"——分类讨论用。
- 给**外部读者**解释生态时——L1/L2/L3 是产业内最广泛理解的框架。

**它的盲区**：

- 对**具身/机器人**场景，它不说明 sandbox（仿真器）和 controller（电机/关节）应该在哪一层。
- L2/L3 边界本身在模糊化（见 §3.4 模式 f），强行二分会把界限不清的项目硬塞进某一档。
- 它把"模型选择"（GPT vs Claude）和"harness 选择"（Pi vs Claude Agent SDK）放在同一根纵轴上，但前者是**横向替换**、后者是**纵向叠加**——视角混淆。

**结论**：这个视角对外解释好用，做选型时**不够精细**。需要配合视角 II–IV。

---

### 4.2 视角 II：按职责（perception / memory / planning / skill / action / communication）

**定义**：把每个 agent 实例分解为 6 个**正交职责**，每个独立可替换、可 ablate、可 benchmark。这是学术综述（"LLM-based Agentic Reasoning Frameworks: A Survey"，arXiv 2508.17692；"Large Language Model Agents: A Comprehensive Survey"，*Preprints* 2025）默认的视角。

| 职责 | 含义 | 在 roboclaws 里 |
|------|------|------------------|
| **Perception** | 把环境状态翻译成 LLM 可消费的形式 | VLM 看 AI2-THOR RGB-D 帧 + 对象列表 + agent 位姿 + 队友位姿 |
| **Memory** | 跨时刻保留信息 | 4 层架构：SOUL.md（不变）+ MEMORY.md（反思）+ episodic FTS（事实）+ archival vector（语义） |
| **Planning** | 高层目标→子目标分解 | 把 territory-control objective 拆成 patrol/claim/defend；用 ReAct/Reflexion/ToT 等 primitive |
| **Skill / Option** | 可复用的中层行为 | `patrol(region)`、`claim(cell)`、`negotiate_handoff(cell, peer)`；Voyager 风格代码 |
| **Action** | 把 skill 翻译成原子操作 | 离散导航 primitive：`controller.step(action="MoveAhead")` 等；code-as-action 形式 |
| **Communication** | agent 间消息传递 | A2A 协议（跨厂商）、ACP（共享消息）、KafClaw 风格 typed Kafka envelope、自定义 blackboard |

**应用到 roboclaws**：

```
              ┌──────────────────────────────────────────────┐
              │  Perception: AI2-THOR RGB + 对象表 + 位姿     │
              └──────────────────────────────────────────────┘
                           ↓
              ┌──────────────────────────────────────────────┐
              │  Memory: SOUL ⇄ MEMORY ⇄ FTS ⇄ vector        │
              └──────────────────────────────────────────────┘
                           ↓
              ┌──────────────────────────────────────────────┐
              │  Planning (VLM): 目标分解 + ReAct loop         │
              └──────────────────────────────────────────────┘
                           ↓
              ┌──────────────────────────────────────────────┐
              │  Skill library (Voyager 风格代码)             │
              └──────────────────────────────────────────────┘
                           ↓
              ┌──────────────────────────────────────────────┐
              │  Action: controller.step (code-as-action)    │
              └──────────────────────────────────────────────┘

              横切：Communication（agent 间消息总线）
```

**它最适合回答的问题**：

- "我们想换 memory 实现，影响哪些代码？"——只动 Memory 一根柱子。
- "做 ablation 时哪个组件最关键？"——分别切除每根柱子做对照实验（Generative Agents 论文就是这么做的）。
- "写论文 / 写技术报告时怎么组织章节？"——按 6 个职责各一节。
- "招贡献者时，哪个职责最缺人？"——按职责切分工作。

**它的盲区**：

- 不说**部署边界**——Memory 是和 agent 同进程？是远程服务？是 agent 自己写？这个视角不区分。
- 不说**时间尺度**——Planning 是每 episode 一次还是每 step 一次？Action 是 1 Hz 还是 1 kHz？要靠视角 IV 补。
- 对**多 agent 协调**着墨不够——Communication 只是其中一根柱子，但 territory-control 这种任务里它可能比 Planning 还重要。

**结论**：做组件级实验、写论文、内部分工时**最好用**。做架构决策时需要配合视角 III、IV。

---

### 4.3 视角 III：按部署形态（CLI / gateway / cloud / embedded）

**定义**：按"agent 在哪里运行、怎么启动、生命周期多长"切层。这是变体格局**实际上**聚类的轴——§3.2 表格里那些 Rust/Zig/Go 重写区分自己的方式。

| 形态 | 生命周期 | 代表项目 | 适用场景 |
|------|----------|----------|----------|
| **短生命 CLI / SDK** | 一次任务，秒-分钟 | Pi、Smolagents、Claude Agent SDK、Aider | 训练时、batch eval、CI 跑 |
| **持久本地 gateway** | 长期运行，天-月 | OpenClaw、Hermes Agent、Nanobot、NanoClaw | 个人助手、demo、研究员实时交互 |
| **云托管 / serverless** | 弹性伸缩，事件触发 | Moltworker（Cloudflare）、E2B、Modal、Hermes-on-Modal | 规模化并行 rollout、Behavior-Best-of-N |
| **嵌入式 / on-device** | 设备生命周期 | PicoClaw、NullClaw、MimiClaw、GR00T-on-Jetson | on-robot 推理、边缘部署 |

**应用到 roboclaws**：

| 当前/未来 | 形态 | 说明 |
|-----------|------|------|
| 单机研究开发 | 短生命 CLI / SDK | 跑实验时，agent 与 sim 同进程或子进程 |
| 实时交互 demo | 持久本地 gateway | 研究员要"对着运行的 episode 说话" |
| 大规模 eval | 云托管 / serverless | Phase-2 里如果要 100 episode × 多 seed × 多 model 跑 grid search，必上云 |
| Phase-3 on-robot | 嵌入式 | G1 humanoid 上跑 GR00T System-1，VLM System-2 可远程 |

**它最适合回答的问题**：

- "这个东西要花多少 GPU/RAM？"——形态决定资源 envelope。
- "我们要不要自建机房还是用 AWS？"——形态决定云成本结构。
- "Phase-3 的 on-robot 部署需要哪些技术？"——直接对应"嵌入式"档。
- "为什么 Rust/Zig 重写这么多？"——它们对应"嵌入式"和"云托管"形态的资源约束。

**它的盲区**：

- 对**内部架构**无话可说——同一个形态下的不同项目，内部可能完全不同。
- 对**职责切分**无话可说——同一个 gateway 项目里，memory 可能内嵌也可能外接。

**结论**：做**运维和成本决策**时最好用。做**代码组织**时不够。

---

### 4.4 视角 IV：具身 agent 栈（reasoner / skill / controller / sim）

**定义**：按"控制频率 + 抽象时间尺度"切层。这是 NVIDIA GR00T、Voyager、SayCan、PaLM-E、RoBridge、HELP 这一谱系隐含的视角，对**具身 AI 项目**最自然。

| 层 | 角色 | 频率 | 时间尺度 | 代表实现 |
|----|------|------|----------|----------|
| **L-Hi — System-2 Reasoner** | 解析指令、长程规划、错误恢复 | 1–10 Hz | 秒-分钟 | VLM (Claude/GPT/Gemini)、Cosmos Reason、GR00T System-2 (Eagle 2.5)、Voyager curriculum |
| **L-Mid — Skill / Option / VLA action expert** | 给定子目标产生连续/中粒度动作 | 10–50 Hz | 100ms-秒 | Voyager skill library、π₀/π₀-FAST/π₀.₅、GR00T System-1 (DiT denoising)、RT-2、OpenVLA、ACT、Diffusion Policy、SmolVLA |
| **L-Lo — Controller** | 关节力矩、平衡、locomotion | 50 Hz - 1 kHz | <100ms | Isaac Lab 训练的 RL whole-body controller、MoveIt、MPC、ROS BehaviorTree |
| **L-Sim — Substrate** | 提供物理/感官世界 | (是 env，不是 controller) | – | AI2-THOR、Isaac Sim、Habitat、ThreeDWorld、真机 |

**应用到 roboclaws**：

```
                   ┌────────────────────────────────────────┐
L-Hi  reasoner     │  per-agent VLM (Claude / GPT / Kimi)    │
                   │  + SOUL.md + MEMORY.md                  │
                   │  + Voyager 风格 skill library           │
                   └──────┬─────────────────────────┬────────┘
                          │ MCP tool calls          │ A2A messages
                          ▼                         ▼
L-Mid skill        ┌───────────────────────┐  ┌──────────────────────┐
                   │  Phase-1/2: Python    │  │  agent 间消息总线     │
                   │  skill (patrol/claim) │  │  (KafClaw 风格)       │
                   │  ────────────────     │  └──────────────────────┘
                   │  Phase-3: π₀.₅ 或     │
                   │  GR00T System-1       │
                   └────────┬──────────────┘
                            ▼
L-Lo  controller   ┌────────────────────────┐
                   │  Phase-1/2: AI2-THOR   │
                   │  controller.step (离散)│
                   │  ─────────────────────  │
                   │  Phase-3: Isaac Lab RL │
                   │  whole-body controller │
                   └────────┬───────────────┘
                            ▼
L-Sim substrate    ┌────────────────────────────────────────┐
                   │  AI2-THOR 今天 → Isaac Sim → 真 G1     │
                   └────────────────────────────────────────┘
```

**Phase 之间的迁移路径在这个视角下最清晰**：

| 层 | Phase 1 (已完成) | Phase 2 (进行中) | Phase 3 (规划中) |
|----|------------------|-------------------|-------------------|
| L-Hi | 单 VLM 直调 | per-agent VLM + SOUL/MEMORY | 同左，可选自部署 LLM |
| L-Mid | 没有显式中层 | Voyager 风格 Python skill 库 | + GR00T S1 / π₀.₅ 接管连续动作 |
| L-Lo | AI2-THOR `step` | 同左 | Isaac Lab RL whole-body |
| L-Sim | AI2-THOR | 同左 | Isaac Sim → 真 G1 |

**它最适合回答的问题**：

- "Phase-3 时哪些模块会被替换？"——精确地按层指出。
- "这个 paper 提的方法属于栈里哪一层？"——VLA 综述就是这个视角的产物。
- "我们的瓶颈在哪一层？"——延迟分析按层做。
- "下一篇要读哪些论文？"——按层选阅读 list（VLM planner 论文、VLA 论文、locomotion RL 论文分开追）。

**它的盲区**：

- **不适合外部读者**——非机器人背景的人看不懂 L-Hi/L-Mid/L-Lo 的命名。
- **memory 在哪一层？**——这个视角不直接讲。MEMORY 横跨 L-Hi 和 L-Mid，需要补充说明。
- **多 agent 通信在哪一层？**——同样横切的。需要把视角 II 的 Communication 拉过来作为补充。

**结论**：做**Phase 迁移规划**和**学术阅读**时最好用。是 roboclaws 文档里"内部用"的核心视角。

### 4.5 4 个视角对比与组合使用建议

| 视角 | 切轴 | 最适合 | 不适合 | 给谁看 |
|------|------|--------|--------|--------|
| I — 抽象层级（L1/L2/L3） | 距离 LLM 原始 API 多远 | 跨生态对比、外部沟通 | 具身决策、Phase 迁移 | 外部读者 |

## 5. 学术与产业的现成分类法

§4 是为 roboclaws 量身切的 4 个视角。本节是**外部参考**——学术界和产业界已经有的分类法。如果你要写论文、对外讲 talk、或对接其他研究组，这些是更标准的"通用语"。

### 5.1 学术综述（2024–2026）

学术界对 LLM agent 的分类法在 2024 年以前还很分散，到 2026 年初已经基本收敛。最值得读的几篇：

**"Large Language Model Agents: A Comprehensive Survey on Architectures, Capabilities, and Applications"** (*Preprints* 2025, [arXiv preprint 形式](https://www.preprints.org/manuscript/202512.2119))

- 提出 **4 轴正交分解**：reasoning enhancement (CoT/ToT/Reflexion) + tool augmentation + memory augmentation + multi-agent collaboration。
- 重点：这是**正交能力**、不是**栈**。同一个 agent 可以在 4 轴上各自打分。

**"LLM-based Agentic Reasoning Frameworks: A Survey from Methods to Scenarios"** ([arXiv 2508.17692](https://arxiv.org/html/2508.17692v1))

- 把 reasoning 模式系统化：ReAct、Reflexion、Tree-of-Thoughts、Multi-agent debate、Generator-Verifier、Plan-then-Execute、Code-as-Action。
- 强调这些**与框架层正交**——你在任意 harness 内组合它们。
- 给多 agent 编排提了 3 形式（中心化/分布式/层级）× 3 模式（合作/竞争/谈判）的 9 宫格。对 roboclaws 的 territory-control（competitive）+ cooperative coverage（cooperative）正好覆盖两端。

**"AI Agent Systems: Architectures, Applications, and Evaluation"** ([arXiv 2601.01743](https://arxiv.org/html/2601.01743v1))

- 提出更工程化的 7 层栈：Foundation Model → Tool Use → Memory → Planning → Multi-Agent → Evaluation → Deployment。
- 与 §5.2 的产业栈图大体吻合。

**VLA 领域的两篇综述**：

- **"A Survey on Vision-Language-Action Models for Embodied AI"** ([arXiv 2405.14093](https://arxiv.org/abs/2405.14093))——较早的 VLA 全景。
- **"Large VLM-based Vision-Language-Action Models for Robotic Manipulation: A Survey"** ([arXiv 2508.13073](https://arxiv.org/html/2508.13073v1))——明确提出 **monolithic vs. hierarchical VLA** 的分类，hierarchical 又分 planner-only 和 planner+policy。是视角 IV 的学术基础。

**多 agent 具身的关键 paper**：

- **Voyager** ([arXiv 2305.16291](https://arxiv.org/abs/2305.16291))——自动 curriculum + 不断成长的 skill library + 迭代 prompt。
- **Generative Agents** ([arXiv 2304.03442](https://arxiv.org/abs/2304.03442))——memory stream + reflection + planning，三组件 ablation 都关键。
- **Two Body Problem (TBONE)** ([arXiv 1904.05879](https://arxiv.org/pdf/1904.05879))——AI2-THOR 多 agent 鼻祖。

**意涵**：roboclaws 的 §6 推荐里给 SOUL/MEMORY/skill library 的设计来自 Voyager + Generative Agents 的 ablation 证据，不是凭直觉。

### 5.2 产业栈图

产业的栈图比学术的更具体、更实用，但散落在各家厂商博客里，没有标准。以下是把多家观点综合后的**产业事实栈**：

```
│                                  Agents SDK、LangGraph、Smolagents  │
├────────────────────────────────────────────────────────────────────┤
│  Layer 4  Sandbox / Runtime   — E2B、Modal、Daytona、Sprites、       │
│                                  Runloop、Northflank Sandboxes      │
├────────────────────────────────────────────────────────────────────┤
│  Layer 3  Tools                — 内置工具 + 外部 CLI                 │
├────────────────────────────────────────────────────────────────────┤
│  Layer 2  Protocol             — MCP（工具）、A2A（agent 间）、      │
│                                  ACP（编辑器/通信）                  │
├────────────────────────────────────────────────────────────────────┤
│  Layer 1  Model / Inference    — Anthropic、OpenAI、Google、         │
│                                  Mistral、Qwen、自部署               │
└────────────────────────────────────────────────────────────────────┘

横切：Memory（Mem0、Letta、Zep、自实现）、Multi-agent 编排（CrewAI、AutoGen、LangGraph）
```

引用来源：

- **a16z "Big Ideas 2026"** —— 提出 6-layer 栈的核心结构。
- **LangChain "Agent ↔ Sandbox connection patterns"** ([docs.langchain.com](https://docs.langchain.com)) —— 强调 Layer 4 与 Layer 5 解耦是 2026 年的方向。
- **Anthropic "Inside Claude Agent SDK"** —— 显式区分 harness 和 sandbox。

- **Modal、Northflank、ZenML 的多篇 sandbox 对比文章** —— 具象 Layer 4 的差异化。
- **NVIDIA GR00T blog** —— 把 Layer 1 的 model 维度细分到 VLA 子领域。

**这套栈和你的 L1/L2/L3 的对应**：

| 你的 | 产业事实栈对应 |
|------|----------------|
| L1 | Layer 1 + Layer 2（一些） |
| L2 | Layer 5 |
| L3 | Layer 5 + Layer 6（绑了 channel） |

所以 **你的 L1/L2/L3 是产业栈的一个粗粒度投影**——它压缩了 Tools/Sandbox/Protocol 的细节。这对外部沟通有用（少 3 层信息量），但做架构决策时会丢信息。

## 6. 给 roboclaws 的具体建议

前 5 章是描述性的（生态长什么样、有哪些视角）。这一章是**规范性的**——基于前 5 章的证据，给出 roboclaws 应该做什么的具体建议。每条建议都标注了证据来源章节，如果未来证据变了，对应建议要重审。

### 6.1 按 phase 的取舍

不要"一步到位选最好的栈"——按 phase 决定。每个 phase 的约束不同，最优解也不同。

**Phase 1 / Phase 2 当前（AI2-THOR + 离散导航 + 多 agent + per-agent SOUL）**：

主推 Path A，作为 fallback 给 Path B。

| Path | L-Hi (reasoner) | L-Mid (skill) | L-Lo (controller) | 评分 |
|------|------------------|----------------|---------------------|------|
| **A — Hermes Agent (Python harness)** | Hermes Python harness 多 provider 支持 + Hermes 内置 episodic memory + SOUL.md | Voyager 风格代码 skill 库 | AI2-THOR step | ★★★★★ |
| B — Smolagents CodeAgent | Smolagents 1k LoC 内核 + 自写 SOUL/MEMORY 加载 | code-as-action 直接生成 step 序列 | AI2-THOR step | ★★★★ |
| C — 自建小 harness + Pi (TS) RPC | 自己写 Python loop，通过 RPC 调 Pi-coding-agent | Pi 的 read/edit/write/bash | AI2-THOR step | ★★★ |
| D — 继续用 OpenClaw Gateway | OpenClaw（保留现状） | OpenClaw skill | AI2-THOR step | ★★（语言阻抗、L3 形态不对路） |

**为什么 A 比 B 略优**：Hermes 的 episodic memory + 自动 skill 生成 + 多 backend（包括 Daytona/Modal serverless）已经做得很扎实，而你在 Phase-2 急需 per-agent memory 这件事 ready-made。Smolagents 内核更小但 memory 要自己造。

**为什么 D 弱**：OpenClaw 是 TypeScript（你 90.8% Python，子进程 IPC 有阻抗痛点），而且它的 L3 形态（持久 gateway 持有 loop）天然不适配你"sim 驱动时间"的 SDK 形态需求。它在 §3.4(f) 提到的 "loop 谁拥有"问题上站错了边。

**Phase 3 计划（Isaac Lab + G1 humanoid + RL locomotion）**：

| L-Hi | L-Mid | L-Lo | 备选 |
|------|-------|------|------|
| 同 Phase 2（Hermes 或 Smolagents）+ MCP 化 Isaac 工具 | **GR00T N1.7** (NVIDIA, 商业 license) **或** **π₀.₅** (openpi, 完全开源) | Isaac Lab 训练的 RL whole-body controller | – |

**GR00T vs π₀ 选择**——这是 Phase-3 最大的未决问题：

| 维度 | GR00T N1.7 | π₀.₅ |
|------|------------|------|
| License | 商业 (2026/4 发布) | 完全开源 (Apache-ish) |
| 预训练数据 | 20.8k 小时 EgoScale 自我视角 + 多源 | LeRobot 数据集 + Physical Intelligence 自家 |
| 与 Isaac 集成 | 一等公民 | 通过 LeRobotDataset v2 兼容 |
| Humanoid 支持 | GR00T-WholeBodyControl + COMPASS | π₀.₅ 跨形态 |
| 与 G1 适配 | 需要数据 finetune | 同 |
| 推理性能 | NVIDIA 优化 | 通过 websocket 远程 |

下次 checkpoint 应该针对这个做专门评估（见 §7）。

### 6.3 工程上立刻可以做的 6 件事

不需要等 Phase-3、不需要换栈，本月就能做：

1. **从今天起所有 rollout 用 LeRobotDataset v2 格式存盘**。即使你现在只是在 AI2-THOR 里跑，数据格式对齐 LeRobot 后，未来 Phase-3 迁到 Isaac/GR00T 时是 zero-cost 的。[LeRobot dataset 文档](https://huggingface.co/docs/lerobot)。
2. **把 AI2-THOR `controller.step` 包成 MCP server**。给每个 sim 实例一个 MCP 端点，agent 通过 MCP 调用而不是直接函数调用。这样未来换任何 harness（Hermes/Smolagents/Claude Agent SDK）都不用改 sim 接口。
3. **每个 robot 写一个 `SOUL.md`**。直接照搬 [openclawsoul.org](https://openclawsoul.org/) 的格式或 OpenPersona 的 schema。包含：性格倾向（保守/激进）、风险态度、与队友的关系、作战风格偏好。
4. **每个 robot 写一个 `MEMORY.md`** 模板，让 agent 在 episode 结束后自己写 reflection 进去。先用 Letta 或 Mem0 作为后端实现，自己造比它们慢。
5. **Skill 用 SKILL.md 格式**。即使现在还没切到 Hermes/Smolagents，也用这个格式（frontmatter + body）。文件里 body 是 Python 代码块，描述 metadata 在 frontmatter。
6. **架构图重画 4 张**——按 §4 的 4 个视角各画一张，放到 README + ARCHITECTURE.md。让不同读者各取所需。

## 7. 公开问题与下次 checkpoint 该回答什么

下次 checkpoint（2026 年 5 月底）应优先回答这些：

**实验性问题**（需要在 roboclaws 自己的代码里测才能回答）：

- ☐ **Q1**：在 AI2-THOR 离散导航场景下，code-as-action 真的比 JSON tool-call 少 30% LLM step 吗？还是 Smolagents 那个数字主要是代码任务？
- ☐ **Q2**：用 Hermes Agent 的自动 skill 生成在多 agent 仿真场景里，新生成的 skill 能跨 agent 复用吗？还是会过拟合到生成它的那个 agent 的 SOUL？
- ☐ **Q3**：4 层 memory（SOUL → MEMORY → FTS → vector）对 territory-control 这种 short-horizon 但 high-frequency 任务，全部 4 层都需要吗？还是只需要前 2 层？

- ☐ **Q4**：Phase-3 的 GR00T N1.7 vs π₀.₅，针对 G1 humanoid 应该选哪个？影响因素：商业 license、数据 finetune 难度、社区活跃度、与 Isaac 的耦合度。
- ☐ **Q5**：Hermes Agent vs Nanobot vs Smolagents 三选一时，跑相同 roboclaws 任务的实测 step 数和正确率。
- ☐ **Q6**：MCP 化 sim 接口的延迟开销有多大？如果 sim step 是 30 Hz，MCP RPC 的 round-trip 是否会成为瓶颈？

**生态跟踪问题**（每月扫一次）：
