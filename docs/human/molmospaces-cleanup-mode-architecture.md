# MolmoSpaces Cleanup Mode Architecture

Status: draft for refactor planning

This note defines the naming model for MolmoSpaces cleanup command modes. It is
intended to guide a later CLI/docs/test refactor without changing behavior by
itself.

## Problem

The current command surface mixes several separate questions:

- What does the agent actually receive as input?
- Where did that input come from?
- Which world backend is running?
- What visual or semantic artifacts are produced for humans?
- Which verifier profile decides pass/fail?

The confusing case is `visual`. In current Molmo cleanup commands, `visual`
means "produce RBY1M robot-view report artifacts." It does not mean the agent is
reasoning over images. The default visual cleanup run still uses structured
object detections.

The names `raw_fpv_only` and `camera_model_policy` are also easy to forget.
They describe current implementation details more than the actual contract.

## Design Goals

- Make the command name reveal what the model sees.
- Keep simulator and real-robot backends under the same public abstraction.
- Avoid names that imply image reasoning when the model receives structured
  labels.
- Avoid names that imply simulator-only behavior when the same input shape could
  later come from a real robot semantic map or detector.
- Keep provenance explicit but separate from the agent input shape.
- Preserve old names as aliases during migration.

## Primary Axis: Agent Input

`agent_input` answers: what information is the decision-making agent allowed to
use?

| Proposed name | Current equivalent | Meaning |
| --- | --- | --- |
| `world_object_labels` | `visible_object_detections` | Structured object handles, categories, support estimates, and candidate fixtures from a world model. No raw image reasoning is required. |
| `raw_camera_image` | `raw_fpv_only` | Raw camera/FPV observation artifacts. No structured object labels are provided by the contract. |
| `camera_object_labels` | `camera_model_policy` | Structured object candidates produced from a camera observation by a perception step. Today this is simulated; later it can be a real detector or VLM. |

### `world_object_labels`

This mode gives the agent handles such as `observed_001`, object categories,
support estimates, and public candidate fixture hints. The source can be a
simulator, a robot semantic map, a maintained object memory, or another world
model. The important contract is that the agent receives structured labels
directly rather than deriving them from pixels.

Example provenance values:

- `simulator_state`
- `robot_semantic_map`
- `object_memory`
- `human_annotation`

### `raw_camera_image`

This mode gives the agent camera observation artifacts and withholds structured
object labels. This is the mode that should be used when the test is meant to
exercise model image reasoning or image-to-text perception.

Expected contract:

- raw observation id
- image artifact references
- waypoint and room context
- no object handles
- no category labels
- no support estimates
- no candidate fixture hints

### `camera_object_labels`

This mode starts from a camera observation and then registers structured object
candidates through a perception step. It is distinct from `world_object_labels`
because the labels are camera-derived, not map/world-derived.

Current implementation note: the existing `camera_model_policy` path uses
deterministic simulated camera-model evidence. It does not call a real VLM or
detector. That should be represented as:

```text
agent_input=camera_object_labels
perception_provenance=simulated_camera_model
```

A future real vision run could use the same input shape with different
provenance:

```text
agent_input=camera_object_labels
perception_provenance=vlm_detector
```

## Supporting Axis: Perception Provenance

`perception_provenance` answers: where did the labels or observations come from?

Recommended values:

| Value | Meaning |
| --- | --- |
| `simulator_state` | Structured labels derived from simulator object locations/state. |
| `robot_semantic_map` | Structured labels from a real robot semantic map or object memory. |
| `simulated_camera_model` | Camera-label candidates generated from public simulator state to mimic a camera model. |
| `vlm_detector` | Camera-label candidates generated by a real VLM or multimodal model. |
| `object_detector` | Camera-label candidates generated by a conventional detector. |
| `human_annotation` | Labels supplied by a human or offline annotation process. |

This axis should be metadata first. It may become a command override only when
there are multiple implementations for the same `agent_input` shape.

## World Backend Axis

`world_backend` answers: which world is being run?

| Proposed name | Current equivalent | Meaning |
| --- | --- | --- |
| `synthetic_contract` | `synthetic` | Cheap contract-shaped backend. |
| `molmospaces_sim` | `molmospaces` | Real upstream MolmoSpaces subprocess backend. |
| `real_robot` | none yet | Future real hardware backend. |

This axis should not imply what the agent sees. A real robot can still provide
`world_object_labels` if it has a semantic map, and a simulator can still provide
`raw_camera_image`.

## Report Axis

`report` answers: what artifacts are generated for humans and checkers?

| Proposed name | Current equivalent | Meaning |
| --- | --- | --- |
| `semantic_report` | `semantic` | Shared cleanup report without robot-view timeline. |
| `robot_view_report` | `visual` | Adds RBY1M FPV/chase/map/verification timeline. This is report evidence, not necessarily model input. |
| `raw_camera_report` | part of `raw-fpv` | Emphasizes raw camera observations in the report. Usually combined with `agent_input=raw_camera_image`. |

Avoid using bare `visual` in new public docs. Prefer `robot_view_report` so it
does not sound like image reasoning by the agent.

## Agent Driver Axis

`agent_driver` answers: who is choosing actions?

Current names are mostly serviceable:

| Proposed name | Current equivalent | Meaning |
| --- | --- | --- |
| `scripted_baseline` | `direct` | Deterministic Python cleanup loop. |
| `mcp_smoke` | `mcp-smoke` | Deterministic ADR-0003 MCP tool smoke. |
| `openclaw_smoke` | `openclaw-smoke` | OpenClaw-shaped smoke without live Gateway. |
| `codex` | `codex-live` | Live Codex CLI over MCP. |
| `claude` | `claude-live` | Live Claude Code over MCP. |
| `openclaw` | `openclaw-live` | Live OpenClaw Gateway path. |

`direct` is vague, but it is less dangerous than the current `visual` ambiguity.
Renaming it can be a lower-priority cleanup.

## Verifier Axis

`verifier` answers: what does pass/fail mean?

Recommended values:

| Value | Meaning |
| --- | --- |
| `contract_only` | Contract shape exists and artifacts are present. |
| `cleanup_success` | Cleanup status, restoration rate, coverage, and disturbance thresholds pass. |
| `robot_view_honesty` | Robot-view timeline exists and waypoint/post-place roles are honest. |
| `image_input_contract` | Raw camera/image input path is actually used and structured labels are withheld when expected. |
| `real_robot_alignment` | Report metadata keeps semantic simulation separate from real robot readiness. |

This axis explains cases like the MiMo trial: cleanup metrics passed, but strict
robot-view honesty failed because the last post-place observe was missing.

## Current Default Mapping

The current public command:

```bash
just task::run molmo-cleanup claude visual
```

should be understood as:

```text
task=molmo-cleanup
agent_driver=claude
agent_input=world_object_labels
perception_provenance=simulator_state
world_backend=molmospaces_sim
report=robot_view_report
verifier=cleanup_success + robot_view_honesty + real_robot_alignment
```

It does not mean:

```text
agent_input=raw_camera_image
```

The current raw-FPV report command:

```bash
just molmo::raw-fpv-report
```

should become conceptually:

```text
agent_input=raw_camera_image
world_backend=molmospaces_sim
report=robot_view_report
verifier=image_input_contract
```

The current camera-model policy path should become conceptually:

```text
agent_input=camera_object_labels
perception_provenance=simulated_camera_model
world_backend=molmospaces_sim
report=robot_view_report
verifier=image_input_contract + cleanup_success
```

## Migration Plan

1. Add new public aliases while keeping old values accepted.
2. Update command help and docs to show `agent_input`, `world_backend`, `report`,
   and `verifier` as separate axes.
3. Update report metadata to show both the new public names and legacy internal
   names during migration.
4. Add contract tests proving that `report=robot_view_report` does not imply
   image input.
5. Add default tests for `agent_input=raw_camera_image` and
   `agent_input=camera_object_labels`.
6. Only after the public surface is stable, consider renaming internal constants.

## Compatibility Aliases

| Legacy value | New interpretation |
| --- | --- |
| `visible_object_detections` | `agent_input=world_object_labels` |
| `raw_fpv_only` | `agent_input=raw_camera_image` |
| `camera_model_policy` | `agent_input=camera_object_labels`, `perception_provenance=simulated_camera_model` |
| `synthetic` | `world_backend=synthetic_contract` |
| `molmospaces` | `world_backend=molmospaces_sim` |
| `semantic` | `report=semantic_report` |
| `visual` | `report=robot_view_report` |
| `raw-fpv` | `agent_input=raw_camera_image`, `report=robot_view_report` |
| `direct` | `agent_driver=scripted_baseline` |

## Missing Default Tests

The refactor should make these gaps visible:

| Test target | Purpose |
| --- | --- |
| `agent_input=world_object_labels` with `report=robot_view_report` | Current default structured cleanup path. |
| `agent_input=raw_camera_image` with `report=robot_view_report` | Proves raw camera input is actually used and structured labels are withheld. |
| `agent_input=camera_object_labels` with `perception_provenance=simulated_camera_model` | Proves camera-derived label path is separate from world-model labels. |
| `report=robot_view_report` with any agent input | Proves report visuals are artifacts, not implicit agent image input. |

## Open Decisions

- Whether `direct` should be renamed to `scripted_baseline` in the public
  facade or only documented as such.
- Whether `raw_camera_report` deserves a separate report value, or whether
  `report=robot_view_report agent_input=raw_camera_image` is enough.
- Whether `perception_provenance` should become a user-facing command argument
  immediately or remain report metadata until multiple real implementations
  exist for one `agent_input` mode.
- How soon to expose `agent_input=camera_object_labels` through live
  Codex/Claude/OpenClaw launchers, since the current live server only exposes
  the structured-label and raw-FPV shapes.
