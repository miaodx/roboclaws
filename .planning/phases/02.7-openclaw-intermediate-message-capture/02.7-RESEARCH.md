# Phase 2.7 — Research

## Question

What do we need to know to plan transcript capture for the shipped single-agent
autonomous OpenClaw loop without reopening the Phase 2.6 MCP architecture?

## Current Runtime Facts

### 1. `start_run()` currently only preserves the terminal body

`roboclaws/openclaw/bridge.py` still treats the autonomous run as one blocking
`POST /v1/chat/completions` call:

- `OpenClawBridge.start_run()` builds one request body, calls
  `self._client.post(...)`, parses the final JSON response, and stores only
  `_extract_content(body)` as `RunResult.final_message`.
- There is no streaming path, no SSE parser, and no structured transcript field
  on `RunResult`.
- The bridge already has the right ownership seam for transcript capture:
  everything about the outer Gateway request lives here, and the example only
  needs to persist whatever the bridge returns.

### 2. The autonomous artifact pipeline is additive-friendly

The shipped autonomous path writes:

- `trace.jsonl` from the host-side MCP server and the example runtime
- `run_result.json` from `examples/openclaw_nav_autonomous.py`
- `summary.json`, `replay.gif`, and `report.html` from
  `scripts/render_autonomous_replay.py`

Important compatibility constraints:

- `tests/fixtures/trace_schema_reference.json` freezes the required top-level
  trace keys to `ts`, `tool`, `event`, and `wallclock_elapsed`.
- `frame_capture` payloads are additive-only, but their required keys are fixed.
- `snapshot_metrics()` is equality-checked, so Phase 2.7 should not modify the
  MCP server's 8-key metrics surface just to carry transcript data.
- `scripts/render_autonomous_replay.py` loads every trace line but only gives
  special treatment to `frame_capture`, `request`, and `response` today. New
  event kinds can be added safely if they keep the top-level required keys.

### 3. The real test seams are already in place

Relevant existing tests are:

- `tests/test_bridge.py` — low-level bridge behavior and content parsing
- `tests/test_bridge_start_run.py` — `start_run()` happy path, timeout, errors,
  workspace reset ordering
- `tests/test_openclaw_nav_autonomous.py` — example wiring, `run_result.json`,
  bootstrap env, teardown
- `tests/test_render_autonomous_replay.py` — summary generation and report HTML

There is no `tests/test_openclaw_bridge.py`; the pre-seeded 02.7 stubs pointed
at a non-existent path and need correction.

## Candidate Capture Paths

### Option A — True `/v1/chat/completions` streaming

What it would mean here:

- The bridge sends the same autonomous kickoff request with `"stream": true`.
- The client reads the response as SSE or chunked lines with `httpx`.
- The bridge extracts assistant-visible text deltas or message fragments as they
  arrive and timestamps them relative to run start.

Why it is attractive:

- It is the only path that can capture messages *before* the request finishes.
- It naturally preserves order and timing.
- It best answers the operator question "what did the assistant say between tool
  calls?" instead of reconstructing after the fact.

Risks and unknowns:

- The pinned Gateway surface must actually emit usable assistant text events on
  the OpenAI-compatible stream path.
- The Gateway may stream tool-call deltas, reasoning fields, or nothing useful
  for user-visible transcript capture.
- Once a streaming request is sent, we cannot safely "retry non-streaming" as a
  fallback without risking duplicate autonomous actions. Mode selection must be
  decided *before* the request starts.

### Option B — Terminal-body capture

What it means here:

- Keep today's non-streaming request path.
- Persist the final visible assistant body and explicitly mark it as
  `source=terminal-body`.

Why it is still useful:

- It is compatible with the current bridge code and with the already shipped
  non-streaming Gateway path.
- It gives the report a truthful answer for runs where no stream is available:
  "only the final body was captured."

Hard limit:

- It cannot recover real intermediate messages or their timing. At best it can
  preserve the terminal body as a fallback transcript entry.

### Planning Recommendation

Phase 2.7 should implement both *contracts* but only one primary *mode*:

- Plan 01 must probe the real Gateway and lock the default mode as either
  `streaming-primary` or `terminal-body-primary`.
- Plan 02 should still preserve terminal-body capture as the fallback artifact
  contract, even if streaming wins.
- The example should expose an explicit transcript mode selector for local
  validation, but the bridge must not auto-retry from stream to body inside a
  live autonomous run.

## Recommended Artifact Contract

### Trace events

Add a new additive event kind to `trace.jsonl`:

```json
{
  "ts": "2026-04-22T15:30:12.123456Z",
  "tool": "assistant",
  "event": "assistant_transcript",
  "wallclock_elapsed": 3.42,
  "source": "stream",
  "content": "<note>observing first</note>",
  "message_index": 0,
  "chunk_index": 4,
  "is_final": false
}
```

Notes:

- `tool` must still be present because the frozen top-level trace contract
  requires it. `tool="assistant"` is the least confusing value.
- `wallclock_elapsed` should be measured by the bridge so the renderer can line
  transcript events up with tool requests and frame captures.
- `source` must be explicit (`stream` or `terminal-body`).
- `is_final` distinguishes a terminal-body fallback line from an intermediate
  stream fragment.

### `run_result.json`

Persist transcript metadata additively:

- `transcript_capture_mode` — the requested mode (`stream` or `terminal-body`)
- `transcript_source` — the actual source used on this run
- `transcript_messages` — ordered list of transcript entries

Keeping the full list here makes report regeneration possible even if a trace is
truncated or transcript events are filtered out in a future tool.

## Report-Side Recommendation

`scripts/render_autonomous_replay.py` should:

- Prefer transcript events from `trace.jsonl`
- Fall back to `run_result.json["transcript_messages"]` if the trace has none
- Preserve the existing `Final Message` panel for backwards compatibility
- Add a transcript panel or log that shows:
  - timestamp
  - source badge (`stream` vs `terminal-body`)
  - content

Old runs with no transcript data should still render exactly as they do today.

## Validation Architecture

### Unit-level

- `tests/test_bridge_start_run.py`
  - streaming collection path
  - terminal-body fallback path
  - timeout/error cases preserve truthful transcript metadata
- `tests/test_bridge.py`
  - transcript parsing helpers for content blocks / reasoning-only fields
- `tests/test_openclaw_nav_autonomous.py`
  - example persists `assistant_transcript` events and `run_result.json`
    transcript metadata
- `tests/test_render_autonomous_replay.py`
  - transcript-bearing traces render correctly
  - transcript-free runs remain readable

### Local-only

- Real Gateway spike to lock the default mode
- Real autonomous run with transcript-bearing artifacts
- Report inspection confirming transcript timing/source lines up with tool log

## Concrete Planning Implications

1. Plan 01 should leave behind a reusable local probe helper under `spike/`,
   matching the Phase 2.6 spike pattern.
2. Plan 02 should touch `bridge.py`, the example, the bridge tests, the
   example test, and the trace schema reference.
3. Plan 03 should touch only the renderer, template, and renderer tests.
4. Plan 04 is local-dev only and should update `docs/openclaw-local.md` with
   transcript expectations instead of rewriting the whole Phase 2.6 MCP section.
