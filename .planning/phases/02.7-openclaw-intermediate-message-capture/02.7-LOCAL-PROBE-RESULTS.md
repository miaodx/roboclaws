## Final shipped verdict

`Plan-01 verdict overturned`

The narrow spike was correct that the pinned Gateway can emit real `delta.content` chunks. It was not sufficient to choose the shipped default for the autonomous loop.

Final winner:

- shipped request mode: `terminal-body`
- real autonomous transcript source on timeout-heavy local runs: `session-store`

Why the earlier `streaming-primary` spike lost in full validation:

- the default-mode rerun gave `5` coherent assistant messages and stayed within the expected request wall-clock envelope
- the stream compare gave `217` fragment rows and overran the effective request wall clock to `493.6s`

## Known limitations

- `final_message` still shows `<wall-clock timeout - no final message>` on these long-running local runs; transcript recovery is additive, not a fake terminal body
- for transcript timing, use `wallclock_elapsed`, not the top-level trace `ts`; the transcript rows are appended after `start_run` returns, so their file-write timestamps cluster near teardown
- `session-store` timestamps are reconstructed from the Gateway's per-session JSONL log after the run ends; that is truthful and aligned enough for replay review, but it is not a live streaming UI
