# Phase 2.7: Autonomous OpenClaw intermediate-message capture — Context

**Gathered:** 2026-04-22
**Status:** Ready for planning
**Mode:** Pre-seeded from local review feedback. The missing capability is concrete: current autonomous artifacts show tool traffic and the final assistant message, but not the intermediate assistant messages. Planning should focus on choosing the right capture seam and landing it additively.

<authoritative_source>
## Authoritative Sources

1. `roboclaws/openclaw/bridge.py` — current `start_run()` behavior is a single blocking `/v1/chat/completions` call that extracts only the terminal assistant body into `final_message`.
2. `examples/openclaw_nav_autonomous.py` — current artifact emission path (`run_result.json`, `trace.jsonl`, replay render handoff).
3. `scripts/render_autonomous_replay.py` and `scripts/templates/autonomous_report.html.j2` — current autonomous artifact/report surface.
4. `.planning/milestones/v1.98-phases/02.6-openclaw-mcp-tools-integration/02.6-CONTEXT.md` plus `02.6-SPIKE-FINDINGS.md` — the shipped MCP architecture that this phase must preserve.
5. `docs/retrospectives/plans/phase-2.5-autonomous-nav.md` — prior planning already raised the question of whether useful response streams exist, but 2.6 shipped without transcript capture.

</authoritative_source>

<domain>
## Phase Boundary

Add transcript visibility to the shipped single-agent autonomous MCP loop without reopening the core architecture.

**In scope (MUST deliver):**
- Probe both candidate acquisition paths against the real Gateway:
  - true streaming from `/v1/chat/completions` if the surface emits intermediate assistant text/events
  - terminal-body/fallback capture if streaming is unavailable or unusable
- Persist intermediate assistant messages additively into autonomous artifacts (`trace.jsonl`, `run_result.json`, and/or adjacent transcript file)
- Render the captured messages in `report.html` with timestamps and enough source metadata to tell how they were obtained
- Keep old traces/renderers working; this is an additive schema/report change

**Out of scope (explicit):**
- Replacing the 2.6 MCP tool surface
- Multi-agent autonomous transcript routing
- Reopening the superseded Phase 2.5 curl/exec architecture
- General chat UI work outside the autonomous replay/report artifacts

</domain>

<decisions>
## Implementation Decisions

### D-01: Compare both capture routes before implementation
The first plan is a real-Gateway capability spike. Do not assume streaming exists just because it is preferred.

### D-02: Streaming is preferred if it is real and stable
If the Gateway exposes usable streaming events for intermediate assistant text on the shipped surface, that is the preferred primary path because it best matches normal chat-platform interaction.

### D-03: Fallback capture must still be evaluated
Even if streaming wins, the phase should document what terminal-body-only capture can and cannot recover. The repo needs to know both limits.

### D-04: Additive-only artifact changes
Existing traces and reports must remain renderable. New transcript data may add keys or companion files, but must not break current artifacts.

### D-05: Preserve single-agent scope
Phase 2.7 is a follow-up on the shipped 2.6 single-agent loop. Multi-agent transcript routing is future work.

</decisions>
