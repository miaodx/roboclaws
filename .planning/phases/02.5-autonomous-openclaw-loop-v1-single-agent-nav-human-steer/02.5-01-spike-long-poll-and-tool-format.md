# Plan 02.5-01: Pre-build spike — long-poll + tool-format de-risk

**Source task in reviewed plan:** T49 (see `docs/plans/phase-2.5-autonomous-nav.md`)
**Owner:** local-dev (requires running Gateway + valid `KIMI_API_KEY`)
**Wall estimate:** ~30 min local probe + ~2 hrs cloud research

## Objective

De-risk the two Gateway unknowns that gate every other plan in this phase.
Question 1: does the OpenClaw Gateway image `2026.4.14` support long-lived
`/v1/chat/completions` calls (~3 min) without an internal watchdog timeout?
Question 2: what is the canonical tool-declaration format for SKILL.md in
this image? Both answers are inputs to T52 (blocking-call design) and
T51 (skill tool block). If either answer invalidates the pull-model shape,
STOP and escalate before any code is written.

## Tasks

<tasks>

<task id="01">
<objective>
Run the long-poll spike: bootstrap a Gateway with TIMEOUT_SECONDS=660 and fire a single long-running /v1/chat/completions call that stays open ~3 minutes.
</objective>

<read_first>
- docs/openclaw-local.md
- scripts/openclaw-bootstrap.sh
- docs/plans/phase-2.5-autonomous-nav.md (T49 step 1)
</read_first>

<action>
1. Bootstrap a Gateway locally with the long per-turn watchdog:
   `TIMEOUT_SECONDS=660 TOKEN=$(./scripts/openclaw-bootstrap.sh)`
   Capture the bearer token from stdout.
2. Fire a single POST against `http://127.0.0.1:18789/v1/chat/completions`
   with `model=openclaw/agent-0`, `Authorization: Bearer $TOKEN`,
   `Content-Type: application/json`, and a user message whose content is
   exactly:
   > "Every 30 seconds, emit the current time in a markdown code block.
   > Do this 6 times (roughly 3 minutes total), then reply DONE."
3. Use `curl -sS --max-time 300 -X POST ...` (NOT --max-time 60) so the
   client does not prematurely kill the connection.
4. Observe three things and record them in scratch/phase25-spike.md:
   (a) Did the call stay open for ~3 min without Gateway-side timeout?
   (b) Did bytes arrive incrementally (streaming) or all at once at the end?
   (c) What is the observed response shape when the agent is "working" with
       no upstream response yet — does the Gateway buffer silently, or does
       it emit partial chunks?
5. One-line verdict at the end: either
   `safe to proceed with T52's blocking /v1/chat/completions design`
   OR
   `blocks: <concrete reason — e.g. Gateway closes idle calls after 90s>`.
</action>

<acceptance_criteria>
- File `scratch/phase25-spike.md` exists and `grep -q '^## Long-poll finding' scratch/phase25-spike.md` matches.
- File contains a literal verdict line beginning with `safe to proceed` OR `blocks:` under the long-poll section.
- The three observations (stays-open, streams-vs-buffers, working-shape) are each present as a labelled line or bullet in the section.
</acceptance_criteria>

<verify>
grep -q "^## Long-poll finding" scratch/phase25-spike.md && grep -Eq "^(safe to proceed|blocks:)" scratch/phase25-spike.md
</verify>

<done>
Long-poll section of scratch/phase25-spike.md answers all three observation bullets with a clear one-line verdict appended.
</done>
</task>

<objective>
Run the tool-format spike: read the Gateway internals doc first, then inspect /app/dist/skill-*.js inside the running container to determine the canonical tool-declaration format.
</objective>

<read_first>

- skills/ai2thor-navigator/SKILL.md
- docs/plans/phase-2.5-autonomous-nav.md (T49 step 2)
</read_first>

<action>
1. First read `docs/openclaw-gateway-internals.md` end-to-end. It is the
   prior art on Gateway image spelunking.
2. With the Gateway from task 01 still running, inspect the skill loader:
   `docker exec openclaw-gateway sh -c 'cat /app/dist/skill-*.js | head -300'`
   If more context needed, widen to `head -800` or grep for `tool` / `tools`
   / `mcp` / `function` / `declare` inside /app/dist/.
3. Answer three concrete questions in scratch/phase25-spike.md:
   (a) What is the declaration mechanism? Pick ONE: free-text SKILL.md prose,
       YAML front-matter block, MCP manifest file, JSON Schema tools array,
       or per-tool scripts deposited in workspace/<agent>/skills/.
   (b) Does the agent invoke declared tools directly (the Gateway parses
       them from SKILL.md and exposes them as tool-call slots), or does the
       skill system wrap them (e.g. requires a generated wrapper module)?
   (c) What does a tool invocation LOOK like at runtime — is it an HTTP
       `curl` from a shell tool the agent has, or a dedicated JSON-RPC /
       MCP call the Gateway routes itself?
4. One-line verdict at the end: either
   `safe to proceed with T51's planned SKILL.md ## Tools block + T50's HTTP contract`
   OR
   `blocks: <concrete reason — e.g. Gateway wraps every tool in MCP, so /observe and /move need an MCP shim, not raw HTTP>`.
5. If the answer says "blocks", note what shim or redesign T50 / T51 would need, and stop before doing further work.
</action>
