# Plan 02.5-04: OpenClawBridge.start_run — kickoff + blocking wait + workspace reset + shared-client timeout isolation

**Source task in reviewed plan:** T52 (see `docs/plans/phase-2.5-autonomous-nav.md`)
**Owner:** cloud
**Wall estimate:** 2-3 hrs

## Objective

Add `start_run()` to `OpenClawBridge`. One long-lived POST to
`/v1/chat/completions` that blocks until the Gateway returns (agent
called `done` → server set `done_event` → Gateway emits final message)
or the wall-clock cap fires. Critically: a per-call timeout
(`wall_budget_s + 60`) MUST be passed into the POST kwarg, NOT written
to the shared HTTP client's `timeout` attribute — otherwise every
push-model `step()` caller inherits a 660s+ timeout and the retry
semantics in OpenClawProvider break. A CRITICAL regression test
enforces this. Also wipes the agent's workspace `state/` dir before
each kickoff so MEMORY from previous runs doesn't poison the new one
(matches logged `[fixed-session-prefix-leaks-memory]` pitfall).

2. Inside class `OpenClawBridge`, add a new method `start_run`
   alongside `step()` and `ping()`:
   ```python
   def start_run(
       self,
       agent_id: int,
       prompt: str,
       wall_budget_s: float,
       done_event: threading.Event,
   ) -> RunResult:
   ```

3. Implementation logic:

   a. **Workspace reset (before POST)** — call
      ```python
      import subprocess
      subprocess.run(
          ["docker", "exec", "openclaw-gateway", "sh", "-c",
           f"rm -rf /home/node/.openclaw/workspaces/{self._agent_prefix}{agent_id}/state/*"],
          check=False,
          capture_output=True,
      )
      ```
      Non-fatal on error (e.g. no container running → CI). Log a warning
      via `print(...)` to stderr on non-zero return code but continue.
      This MUST happen BEFORE the POST so the agent starts each run with
      a clean state/ dir. Container name is hard-coded to
      `openclaw-gateway` to match scripts/openclaw-bootstrap.sh's default
      CONTAINER env.

   b. **Blocking POST with per-call timeout**:
      ```python
      import time, httpx
      started = time.monotonic()
      payload = {
          "model": self.model_id(agent_id),
          "messages": [{"role": "user", "content": prompt}],
          "max_tokens": _DEFAULT_MAX_TOKENS,
      }
      per_call_timeout = wall_budget_s + 60
      try:
          resp = self._client.post(
              _CHAT_PATH,
              json=payload,
              timeout=per_call_timeout,  # CRITICAL: kwarg, not attribute mutation
          )
      except httpx.ReadTimeout:
          wallclock = time.monotonic() - started
          return RunResult(
              final_message="<wall-clock timeout — no final message>",
              wallclock_s=wallclock,
              terminated_by="wall_clock",
          )
      except httpx.ConnectError as exc:
          raise OpenClawUnavailable(f"Gateway unreachable: {exc}") from exc
      ```

   c. **Error-status handling**: mirror `_post_chat` — 401 / 404 / 400 /
      5xx all raise `OpenClawUnavailable`. Do NOT catch these and
      convert to `terminated_by="error"` at this layer; let the caller
      decide. Only `httpx.ReadTimeout` maps to `terminated_by="wall_clock"`.

   d. **Parse response** on success:
      ```python
      body = resp.json()
      final_message = _extract_content(body)
      wallclock = time.monotonic() - started
      return RunResult(
          final_message=final_message,
          wallclock_s=wallclock,
          terminated_by="done",
      )
      ```
      We do NOT need to wait on `done_event` here — if the agent truly
      called /done, the sim_server's handler set the event, AND the
      Gateway returned the agent's final message. We trust the Gateway's
      return as the signal. The `done_event` parameter is preserved in
      the signature for future expansion (e.g., early termination when
      sim-side observes an unrecoverable error even though the Gateway
      hasn't returned yet) and for symmetry with the sim_server API.

   e. **NO RETRY**: unlike `step()`'s `_RETRY_ATTEMPTS=3` loop, a
      long-running run has no "try again in 8s" semantics. One shot.

4. **Preserve the existing shared client's timeout**. The constructor
   keeps doing `httpx.Client(..., timeout=timeout)` where `timeout`
   defaults to 180. No change to that line. The per-call timeout on
   start_run is supplied ONLY via the `timeout=` kwarg on the `.post()`
   call — httpx treats that as a per-request override without touching
   the client's default.

5. **Type annotations + ruff**: all public signatures annotated;
   `from __future__ import annotations` already at top of file.
   Line length 100.

## Rescue Paths

| Failure | Rescue |
|---|---|
| Kickoff call exceeds `wall_budget_s + 60` | httpx.ReadTimeout fires → start_run returns `terminated_by="wall_clock"` with a best-effort partial `final_message`. Server stays up. |
| Agent never calls done and never moves (stuck in reasoning) | Wall-clock cap fires. No rescue — this is a quality regression, not a crash. Investigate via trace.jsonl: if `tool_calls == 0`, the skill declarations didn't land (revisit Plan 02.5-03 task 03). |
| Memory accumulation across runs on long-lived Gateway | Covered by the workspace reset in step 1a of task 01 above. Regression-tested by `test_start_run_wipes_workspace_state_before_post`. Matches logged `[fixed-session-prefix-leaks-memory]`. |
| Shared-client timeout gets mutated to wall_budget+60 and breaks step() | Covered by `test_start_run_does_not_mutate_shared_client_timeout`. If this test fails, revisit task 01 step 3b — the `timeout=` kwarg on `.post()` must NOT be accompanied by any assignment to `self._client.timeout`. |
| subprocess.run fails because docker isn't installed (CI environment) | Non-fatal — the `check=False` kwarg means we log a warning but continue. Tests patch `subprocess.run` to avoid depending on a real Docker. |
