## Fixed Issues

### WR-01: Trace-file writes can race with close(); close does not gate further writes

**Files modified:** `roboclaws/openclaw/mcp_server.py`, `tests/test_openclaw_mcp_server.py`
**Commit:** `22d1f91`
**Applied fix:**
- `close()` now flips `self._closed = True` and closes `self._trace_fp` **inside
  `self._trace_lock`**, so no writer can be mid-write when the file descriptor
  disappears.
- `_write_trace` adds an early-bail `if self._closed: return` plus an in-lock
  re-check to close the race window where `close()` flips `_closed` after we
  read it but before we acquire the lock.
- Added `test_write_trace_after_close_is_safe` as a regression guard —
  exercises `write_runtime_event` + `enqueue_human_message` after `close()`
  and asserts no exception is raised.
- Result: the watchdog + stdin threads (which join with a 0.2s timeout and
  can leave writers in flight) can no longer raise
  `ValueError: I/O operation on closed file` during teardown.

### WR-02: Diagnostic subprocess calls have no timeout; hung `docker exec` stalls teardown

**Files modified:** `examples/openclaw_nav_autonomous.py`, `tests/test_openclaw_nav_autonomous.py`
**Commit:** `2c8a730`
**Applied fix:**
- `_run_capture` now takes `timeout: float = 15.0` (module-level constant
  `_DIAGNOSTIC_CAPTURE_TIMEOUT_S`) and forwards it to `subprocess.run`.
- On `subprocess.TimeoutExpired` the function returns `(124, stdout,
  stderr + "<diagnostic command timed out after Ns>")`. Exit code 124
  matches coreutils `timeout(1)` so operators grepping logs recognise the
  failure mode. Stdout/stderr captured before the timeout are preserved
  (with bytes→str decoding for safety).
- Added two regression tests:
  - `test_run_capture_applies_timeout_and_surfaces_timeout_exit_code` —
    asserts the `timeout=` kwarg is forwarded and `TimeoutExpired` returns
    rc=124 with a human-readable stderr suffix.
  - `test_run_capture_has_default_timeout` — asserts a non-None default
    timeout is always passed, so a future caller that forgets to supply
    one still gets the cap.
- Result: worst-case teardown when the Gateway container is wedged is now
  bounded to ~1 minute (4 commands × 15s), instead of unbounded.

### WR-03: Module docstring + default threat model contradict the only production caller

**Files modified:** `roboclaws/openclaw/mcp_server.py`, `tests/test_openclaw_nav_autonomous.py`
**Commit:** `33e7372`
**Applied fix (both options from the review — docs + test):**
- Module docstring rewritten to explicitly document the Linux + Docker 29.x
  caveat: `host.docker.internal` resolves to the bridge gateway (172.17.0.1)
  and cannot reach the host's 127.0.0.1, so callers on that topology must
  override to `host="0.0.0.0"`. References spike 02.6-06 for evidence and
  states that LAN-exposure risk is accepted for single-operator local-dev.
- `_DEFAULT_HOST` comment updated to match, with a direct pointer to the
  new regression test by name (so anyone "fixing" the default has the
  test to explain why it matters).
- Integration-shape assertion added to
  `test_run_autonomous_navigation_offline_happy_path` (per the review's
  exact suggestion):
  ```python
  _, mcp_kwargs = mcp_factory.call_args
  assert mcp_kwargs.get("host") == "0.0.0.0", (
      "example must override host to 0.0.0.0 on Linux (spike 02.6-06); ..."
  )
  ```
- Result: the docstring now matches the production reality, and reverting
  the example's `host="0.0.0.0"` override to the default flips the test
  visibly red instead of breaking Linux local-dev silently.
