## Phase Boundary

Rebuild the autonomous OpenClaw loop on a **first-class MCP tool surface** so the agent's only path to the AI2-THOR engine is three named tools, with every shell/image/fs escape hatch removed.

**In scope (MUST deliver):**
- A roboclaws-hosted MCP server exposing three tools: `observe`, `move(direction, reason)`, `done(reason)` — over streamable-http, backed by the live AI2-THOR engine.
- Bootstrap seeds `mcp.servers.<name>` and `agents.list[<n>].tools.profile = "minimal"` in `openclaw.json` **before first container start** (avoids the SIGUSR1-restart dance).
- Thin `skills/ai2thor-navigator/SKILL.md` that describes the three tools and the observe→think→move loop in ≤~25 lines — no curl recipes, no exec instructions, no `/tmp` warnings.
- `examples/openclaw_nav_autonomous.py` kickoff-prompt block shrinks to a handful of lines describing the loop; curl/exec instructions deleted.
- Human-interjection stdin path preserved: the MCP `observe` tool drains the interjection queue and surfaces it in the tool result (either as a field in the result JSON or as a second text content block).
- Trace + `report.html` with 👁/🚶 frame badges carried forward from 2.5 (architecture-agnostic — applies to MCP and HTTP runs alike).

**Out of scope (explicit):**
- Multi-agent MCP routing (one agent, one MCP server in v2 — multi-agent comes later).
- Tool retries/back-pressure beyond what FastMCP provides out of the box.
- Turning the existing HTTP `sim_server.py` into a dual-mode server. It either stays as-is (frozen, unused by 2.6) or gets deleted — planner picks the cleaner option during planning.
- `profile: coding` or `profile: messaging` variants.
- Replacing push-model code paths used by Phase 2.2 territory/coverage games. This phase is additive.

<decisions>
## Implementation Decisions (LOCKED)

### D-01: Tool surface — three first-class MCP tools
- `observe() -> { fpv: ImageContent, overhead: ImageContent, state: { position, yaw, scene, step, budget_remaining, human_message?: str } }`
- `move(direction: str, reason: str) -> { position, yaw, step, result: "ok"|"blocked"|"error", error?: str }`
- `done(reason: str) -> { final: true, total_moves, elapsed_s }`
- Tool names as registered by the MCP server. Gateway will expose them to the agent as `<server-prefix>__observe`, `<server-prefix>__move`, `<server-prefix>__done` (spike confirmed this prefix convention).
- No generic "read file" / "execute shell" capability given to the agent.

### D-02: Transport — streamable-http on the host, port 18788
- Roboclaws process hosts the MCP server in-process via FastMCP (Python SDK already in the venv as of spike).
- Gateway container reaches it via `http://host.docker.internal:18788/mcp`.
- Why streamable-http (not stdio): engine lives in roboclaws, Gateway is a consumer — running a stdio MCP child inside the Gateway container would require mounting the engine or proxying, both worse.
- **Config shape** (gotcha from spike — valid shape for Gateway's MCP loader):
  ```json
  "mcp": {
    "servers": {
      "roboclaws": {
        "transport": "streamable-http",
        "url": "http://host.docker.internal:18788/mcp"
      }
    }
  }
  ```
  The key is `transport` (not `type`). Accepted values are `"streamable-http"` or `"sse"`. Invalid keys silently fail at load time with `[bundle-mcp] failed to start server ...: SSE error: Non-200 status code (400)`.

### D-03: Tool allowlist — `profile: "minimal"`, set per agent
- `agents.list[<n>].tools = { "profile": "minimal" }` in `openclaw.json`.
- Spike confirmed: `minimal` reduces the agent's tool list to `session_status` + MCP tools. No `exec`, no `read`/`write`/`edit`, no generic `image`, no `web_fetch`, no `browser`.
- Side benefit: per-turn prompt overhead dropped from 15,396 → 6,285 tokens in the spike (~59% reduction). Keeps Kimi comfortably inside its context + timeout budget.

### D-04: Bootstrap seeds MCP config before first container start
- `scripts/openclaw-bootstrap.sh` writes both `mcp.servers.<name>` and `agents.list[<n>].tools.profile = "minimal"` into `openclaw.json` **before** the container starts.
- Rationale: adding `mcp.servers` to a running Gateway triggers SIGUSR1 → PID-1 exit → container stop. In Docker this means manual `docker start`. Seeding up-front avoids that round-trip.
- The existing `SIM_SERVER_URL` / `--add-host` plumbing is repurposed: `--add-host=host.docker.internal:host-gateway` is still needed (same reason as Phase 2.5 — host-side server in roboclaws process). Port env renames from `SIM_SERVER_URL` to `ROBOCLAWS_MCP_URL` (or similar — planner's call) for clarity.

### D-05: MCP server lives under `roboclaws/openclaw/mcp_server.py`
- New module, **not** a dual-mode extension of `sim_server.py`.
- Exposes `make_roboclaws_mcp(engine: MultiAgentEngine, agent_id: int, ...) -> FastMCP` factory.
- Server owns the frame-capture trace (per-tool-call timestamps, frames, direction, reason, human_message echo).
- Planner decides whether `roboclaws/openclaw/sim_server.py` stays as a frozen compatibility fallback or gets deleted in this phase; default is **delete** once the MCP path is green on local-dev validation (D-08 probes).

### D-06: SKILL.md shrinks to ≤25 lines
- Current `skills/ai2thor-navigator/SKILL.md` (245 lines) enumerates curl recipes, exec patterns, image-tool caveats.
- Rewrite as a short SKILL.md with these sections only: role (one paragraph), tool catalog (names + one-line purposes), loop pattern (observe → think → move, stop condition), interjection protocol (how `human_message` appears in observe results).
- No "don't use /tmp", no "don't use exec", no curl recipes — those tools simply don't exist in the agent's surface anymore.
- SOULs (`aggressive/defensive/cooperative`) remain unchanged.

### D-07: Kickoff prompt in `examples/openclaw_nav_autonomous.py` shrinks too
- Current prompt at lines 52-89 is a 38-line curl/exec cookbook.
- Replace with ~5 lines: "You are navigating an indoor room. Use the observe/move/done tools. Budget: N moves, M seconds. Loop until done or budget-exhausted."
- Delete all references to `curl`, `exec`, `image tool`, `data:image URL`, `workspace media`, `/tmp`.

### D-08: Live-probe gate — container → host MCP + agent vision check
- Before merge, the local-dev validation MUST include at least:
  1. **MCP reachability probe**: from inside the `openclaw-gateway` container, POST `initialize` to `http://host.docker.internal:18788/mcp` and confirm 200 + server info.
  2. **Observe-vision probe**: one `/v1/chat/completions` call that asks the agent to call `observe` and describe the FPV frame contents; confirm the agent reports something consistent with the AI2-THOR scene (not "image omitted", not "I cannot see").
  3. **Tool allowlist probe**: ask the agent to list its tools; confirm `exec`, `image`, `read`, `write`, `browser`, `web_fetch` are all absent.
  4. **End-to-end run**: `python examples/openclaw_nav_autonomous.py --scene FloorPlan201 --max-moves 30 --wall-budget 180` finishes with ≥1 move, no `exec`/`curl`/`image` in the Gateway log, valid `replay.gif` + `report.html`.
- Per user feedback `feedback_live_probe_gate.md`: these probes run against the real Gateway + real Kimi at local-dev, not mocked.

### D-09: Backward compatibility — none required
- Phase 2.5 never executed; no shipped code depends on its contract.
- Territory/coverage games (Phase 2.2) use the push model and are untouched.
- Bridge's `step()` signature stays stable; `start_run()` is new but additive.

### D-10: Human interjection — carried forward from 2.5 decisions
- Stdin reader thread on the host (in `openclaw_nav_autonomous.py`) enqueues interjections into a shared queue.
- The MCP `observe` tool drains the queue and includes any pending message as `state.human_message` in the result.
- The agent's SKILL.md instructs: if `human_message` is present, acknowledge it in the next move's `reason` field and do not call `done` until at least one follow-up action.

<deferred>
## Deferred Ideas

- **Multi-agent MCP routing.** Per-agent MCP scoping (agent-0 sees its own observe/move tools, agent-1 sees a different set). v2 is single-agent; multi-agent is a future phase.
- **MCP over stdio.** Could move the MCP server from host-streamable-http into a stdio child launched by the Gateway. Would avoid `host.docker.internal` entirely but requires AI2-THOR inside the container. Not worth the complexity until multi-agent or cloud-Gateway use cases demand it.
- **Per-tool rate limiting, retries, circuit breakers.** FastMCP defaults are fine for single-agent local-dev.
- **MCP server for the push-model games** (territory, coverage). Phase 2.2 push-model works; don't break it.
- **Image-cache tool** (agent asks for "last-N frames"). The `observe` tool always returns fresh frames; caching is YAGNI.
- **`sessions_*` tool suite.** `profile: minimal` excludes these; if multi-agent coordination needs them later, reintroduce as an `alsoAllow` list, not by broadening the profile.
- **Schema-push gate.** No DB schema in this phase — skip.
- **UI work.** `report.html` is carried forward; no new UI concepts in 2.6.

</deferred>
