# Phase 2.6 â€” Spike Findings

**Goal of the spike:** de-risk Fork B (MCP tool surface) before planning. Two unknowns blocked commitment:

- **U1** â€” Does MCP `ImageContent` in a tool result flow through to the Kimi multimodal input, or does the Gateway downconvert it to text like `[image:png]`?
- **U2** â€” Can we restrict the agent's tool list so it literally cannot fall back to `exec`/`image`/`curl`-via-shell?

**Verdict:** Both YES. Architecture is safe to plan around.

## U1 â€” MCP ImageContent â†’ Kimi multimodal

### Probe design

`spike/mcp_image_probe.py` â€” 55-line FastMCP server, streamable-http on port 18790, one tool `spike_get_red_square()` that returns:
- A text block: `"SPIKE_MARKER_A7K: the attached image contains a single solid color. Identify the color in one word."`
- A 96Ă—96 solid red PNG (RGB 220,20,20) as `ImageContent`.

Registered into a running Gateway via `openclaw mcp set roboclaws-spike '{"transport":"streamable-http","url":"http://host.docker.internal:18790/mcp"}'`.

### Result

`POST /v1/chat/completions` asking the agent to call the tool and reply with a compact JSON.

Agent response (verbatim):
```json
{"color":"red","saw_image":true,"marker_seen":true}
```

MCP probe-server log:
```
Processing request of type ListToolsRequest
Processing request of type CallToolRequest
spike_get_red_square called; returning 272-byte PNG
```

### Conclusion

- Gateway's `sanitizeToolResultImages` pipeline in `/app/dist/tool-images-*.js` preserves `{type: "image", data: base64, mimeType}` blocks (resizes to 1200px / 5MB but keeps them).
- The Anthropic-messages adapter (`/app/dist/anthropic-vertex-stream-*.js` lines 5830-5886) converts those to `image_url: data:${mimeType};base64,${data}` for upstream.
- Kimi's coding-tier plugin path via `anthropic-messages` accepts multi-image; this is the same path that direct multimodal chat used in the Phase 2.5 retro (`docs/retrospectives/openclaw-kimi-provider-debug-2026-04-21.md` lines 92-98).
- The `sanitizeToolResult` function that strips image `data` (in `/app/dist/attempt.tool-run-context-*.js`) is the **audit/logging** path, not the provider-facing path.

**The contract that carried the red PNG to Kimi is the same one that carries FPV + overhead images to the agent in Phase 2.6.**

## U2 â€” `profile: minimal` restricts the tool surface

### Probe design

Patched the running Gateway's `openclaw.json` to set `agents.list[0].tools = { "profile": "minimal" }`, then asked the agent: *"List EVERY tool you have access to, one per line, lowercase, no descriptions. Then print TOTAL=<N>."*

### Result

Before (default `coding` profile â€” same setup used in Phase 2.5 runs):
```
read, edit, write, exec, process, web_fetch, browser, canvas, nodes, cron,
message, gateway, agents_list, sessions_list, sessions_history, sessions_send,
sessions_yield, sessions_spawn, subagents, session_status, tts, memory_search,
memory_get, image, pdf
```
(25 tools)

After (`profile: minimal` + our MCP server):
```
session_status
roboclaws-spike__spike_get_red_square
TOTAL=2
```

### Conclusion

- `cfg.agents.list[<n>].tools.profile` accepts `"minimal" | "coding" | "messaging"`; source of truth is `/app/dist/tool-policy-CwZyJfFc.js` lines 352-354.
- `minimal` yields exactly two baseline capabilities (`session_status`, `update_plan`) plus whatever MCP tools your servers expose.
- There is **no exec, no shell, no fs read/write, no image tool, no web fetch, no browser** available to the agent under `minimal`. Prompt-steering becomes unnecessary â€” the tools literally don't exist in the agent's surface.

### Bonus â€” context-overhead win

Prompt tokens per turn:
- `coding` profile: **15,396 tokens**
- `minimal` + 1 MCP tool: **6,285 tokens**

That's a **59% reduction** on the system-prompt overhead. Material for Kimi's long-horizon autonomous runs â€” more budget for actual reasoning + multi-step interjection without hitting the gateway's idle watchdog.

## Free findings (worth enshrining in docs + planner attention)

### F-1: MCP config key is `transport`, not `type`

Stored shape that **works**:
```json
{ "transport": "streamable-http", "url": "http://host.docker.internal:18790/mcp" }
```

Stored shape that **silently breaks**:
```json
{ "type": "http", "url": "http://host.docker.internal:18790/mcp" }
```

Failure mode when the key is wrong or the transport name is unsupported: Gateway logs `[bundle-mcp] failed to start server "<name>" (<url>): Error: SSE error: Non-200 status code (400)` and the agent simply doesn't see the tools. Agent replies truthfully: *"I don't have access to a tool named `<name>`."*

Source code: `/app/dist/pi-bundle-mcp-tools-CxZ16DeR.js:128-170`. Accepted transport values: `"sse"` or `"streamable-http"`. The `openclaw mcp set` CLI does **not** validate; it writes whatever JSON you give it straight to config.

### F-2: Tool-name prefix convention

Tools registered by an MCP server named `"roboclaws"` with an internal tool name `observe` appear to the agent as `roboclaws__observe`. Double-underscore separator. Plans should refer to tools by the prefixed name when writing acceptance criteria (e.g., "Gateway log contains `roboclaws__observe` call").

### F-3: Changing `mcp.servers` triggers a full Gateway restart

Observed in Gateway log:
```
[reload] config change detected; evaluating reload (meta.lastTouchedAt, mcp)
[reload] config change requires gateway restart (mcp)
[gateway] signal SIGUSR1 received
[gateway] received SIGUSR1; restarting
[gateway] restart mode: full process restart (spawned pid 174)
```

In Docker, PID-1 restart = container exit (status: `Exited (0)`). Recovery requires `docker start openclaw-gateway`. The bootstrap script should **seed `mcp.servers` + `tools.profile` into `openclaw.json` before first container start**, not via `openclaw mcp set` after. Saves a restart round-trip on every `./scripts/openclaw-bootstrap.sh` invocation.

### F-4: FastMCP Python server â€” what works

```python
from mcp.server.fastmcp import FastMCP, Image
mcp = FastMCP("roboclaws", host="0.0.0.0", port=18788)

@mcp.tool()
def observe() -> list:
    """Return FPV + overhead images + structured state."""
    return [
        "<state-json-as-text>",
        Image(data=fpv_png_bytes, format="png"),
        Image(data=overhead_png_bytes, format="png"),
    ]

if __name__ == "__main__":
    mcp.run(transport="streamable-http")
```

- `Image(data=<bytes>, format="png")` â€” pass raw bytes, not base64 string. The SDK encodes internally.
- Python list of `str` + `Image` is the natural return shape for mixed content.
- streamable-http exposes the server at `http://<host>:<port>/mcp` (path is `/mcp`, not `/`).

### F-5: Container-to-host networking

The Gateway container must have `--add-host=host.docker.internal:host-gateway` passed at `docker run` time (already present in `scripts/openclaw-bootstrap.sh` for Phase 2.5). No new Docker flags needed for 2.6.
