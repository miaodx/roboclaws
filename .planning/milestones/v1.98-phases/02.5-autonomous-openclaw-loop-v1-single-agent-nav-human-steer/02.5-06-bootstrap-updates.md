<action>
Edit `scripts/openclaw-bootstrap.sh`:

1. In the `mount_args=(...)` array used by `docker run -d --name
   "$CONTAINER" "${mount_args[@]}" "$IMAGE"` (around line 489-507 of
   the current file), add the flag:
   ```bash
   --add-host=host.docker.internal:host-gateway
   ```
   Place it after the `-p ... -v ...` entries and before the `-e ...`
   entries, so the array reads:
   - port binding
   - volume mount
   - --add-host
   - env vars

   documented only in docs/openclaw-local.md per T53-bis step 4 and the

   resolves `host-gateway` to the Docker bridge gateway IP; macOS and
   Windows Docker Desktop always resolve `host.docker.internal`
   natively. The flag is safe to set unconditionally.

4. Do NOT remove or alter any existing entry in `mount_args`. The
   change is purely additive.
</action>

<action>
Append a new section to `docs/openclaw-local.md`. Title:
`## Autonomous loop — sim server networking (Phase 2.5)`.

Content (expand around these bullets; match the file's existing prose
tone):

1. The autonomous loop (`examples/openclaw_nav_autonomous.py`) runs a
   tool HTTP server on the host at `http://127.0.0.1:18788`. The
   Gateway container must reach this server from inside the container.

2. **Default routing: `--add-host=host.docker.internal:host-gateway`**
   added by the bootstrap unconditionally. Works on Docker ≥20.10 on
   Linux (resolves `host-gateway` to the bridge gateway IP) and on
   macOS / Windows Docker Desktop natively.

3. **Env: `SIM_SERVER_URL`** — defaults to
   `http://host.docker.internal:18788`. The bootstrap exports this
   into the container so the skill's tool declarations in
   `skills/ai2thor-navigator/SKILL.md` can reference a single env-driven
   URL. Override if running the sim server on a non-default port.

4. **Linux-only fallback: `--network host`**. If `--add-host` does not
   resolve on a specific host (some older Linux kernels, SELinux edge
   cases), edit `scripts/openclaw-bootstrap.sh` locally to replace
   `-p "${HOST_IP}:${PORT}:18789"` + `--add-host=...` with
   `--network host`. Note: `--network host` overrides port mapping;
   the Gateway binds directly to `18789` on the host. Not portable —
   Docker Desktop on macOS / Windows does not support it.

5. **Pre-flight live-probe gate**: before running any real session,
   validate the routing with:
   ```bash
   docker exec openclaw-gateway curl -sf \
       http://host.docker.internal:18788/observe | head -c 200
   ```
   A 200 response with JSON is the green light. This is the Plan 02.5-08
   step 1 gate per `feedback_live_probe_gate.md`.

Preserve all existing content in docs/openclaw-local.md. The new section
goes at the end, or immediately before any final "Troubleshooting"
section if one exists.
</action>

## Rescue Paths

| Failure | Rescue |
|---|---|
| Gateway container can't reach `host.docker.internal:18788` on the user's machine | Plan 02.5-08 step 1 is the gate. Default setup uses `--add-host=host.docker.internal:host-gateway` (Docker 20.10+). If that doesn't resolve, fall back to `--network host` (Linux-only) — documented in `docs/openclaw-local.md` per task 03 above. |
| Bootstrap script fails bash -n after edits | Revert task 01 / 02 changes one at a time with `git diff scripts/openclaw-bootstrap.sh`; common cause is mismatched quote or a stray newline in mount_args. |
| User runs a Docker version < 20.10 where `host-gateway` isn't resolved | Surface the fallback in the first troubleshooting block of docs/openclaw-local.md (added in task 03). Live-probe gate fails fast on this — Plan 02.5-08 step 1 catches it. |
